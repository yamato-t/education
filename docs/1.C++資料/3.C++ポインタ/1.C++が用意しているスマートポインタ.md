# **C++ におけるスマートポインタ** 

---
## **代表的なスマートポインタ**

C++ でスマートポインタというと、  
まず下記二つが挙げられる。

- unique_ptr
- shared_ptr

加えて、特殊なポインタとして下記も挙げる

- weak_ptr

---

## **unique_ptr**
**確保したメモリの所有権を、他の誰にも持たせたくない**場合に利用されるスマートポインタ。  

```c++
#include <iostream>
#include <memory>

// メイン関数
int main()
{
	{
		// 確保した int 型メモリ領域の所有権を唯一持つ事が出来るスマートポインタ
		std::unique_ptr<int> ptr(new int);

		// ptr を介してメモリの内容を書き換える
		*ptr = 3;

		// ptr を介してメモリの内容を取得する
		int a = *ptr;
		std::cout << a << std::endl; 
	}
	// ptr が破棄されると new したメモリ領域の所有権を放棄する
	// メモリ解放が行われるので、メモリリークは起こらない
	
	return 0;
}

```

アスタリスクをつけて生ポインタのような使い方をしているが、  
アドレスを直接扱っている訳ではなく、あくまで ```ptr``` を介して、  
アドレスの内容を取得、代入している。

```ptr``` の破棄に合わせて所有権を放棄する。  
`new` で確保したメモリ領域の所有権を持っている唯一のオブジェクトである為、  
同時にメモリの解放も行われる。

### **インスタンス化の方法**

`new` キーワードを利用せず、  
次の様に ```std::make_unique``` を利用してメモリ確保を行う事も出来る。

```c++
// メモリを確保し、所有権を得る
std::unique_ptr<int> ptr = std::make_unique<int>();

// 自作クラス型で引数付きコンストラクタを利用したい場合、最後の()の中に指定する
std::unique_ptr<Animal> animalPtr = std::make_unique<Animal>("動物", 4);
```

### **所有権の放棄**

所有権の放棄は、途中で行う事も可能。

```c++
//　メモリを解放する（所有権もリセットできる）
ptr.reset();
```

スマートポインタから生ポインタを得る事も出来る。  
ただこういった使用をすると、スマートポインタの利点がなくなる為、  
基本的に推奨されない。

```c++
// 生ポインタを取得する
int* temp1 = ptr.get();

// 所有権を放棄して、生ポインタを取得する
// 所有権を放棄した為、破棄された時にメモリが解放されない
int* temp2 = ptr.release();
// 生ポインタを直接 delete する必要がある
delete temp2;
```

### **unique_ptr の特徴**

```unique_ptr``` は、他のスマートポインタに内容を複製する事が出来ない。

```c++
// 複製する事はできない
std::unique_ptr<int> copy = ptr; // コンパイルエラー
```

それは複製によって所有権が「共有」されてしまう事態を防ぐ為。

```unique_ptr``` は「確保したメモリの所有権を持てる唯一のオブジェクト」である為、  
所有権の「共有」は許可されない。

ただ「譲渡」は出来る。  
ただし、その為には**右辺値参照**や**ムーブ**という概念を知る必要がある。

## **shared_ptr**
**確保したメモリの所有権を、他と共有したい**場合に利用されるスマートポインタ。  
基本的な使い方は ```unique_ptr``` と同じ。

```c++
#include <iostream>
#include <memory>

// メイン関数
int main()
{
	{
		// メモリを確保し、所有権を得る
		std::shared_ptr<int> ptr(new int);

		// ptr を介してメモリの内容を書き換える
		*ptr = 10;

		// ptr を介してメモリの内容を取得する
		int a = *ptr;
		std::cout << a << std::endl;

	}
	// ptr が破棄されると new したメモリの所有権を放棄する
	// メモリ解放が行われるので、メモリリークは起こらない
	
	return 0;
}
```

### **インスタンス化の方法**

new キーワードを利用せず、  
次の様に ```std::make_shared ```を利用してメモリ確保を行う事も出来る。

```c++
// メモリを確保し、所有権を得る
std::shared_ptr<int> ptr = std::make_shared<int>();

// 自作クラス型で引数付きコンストラクタを利用したい場合、最後の()の中に指定する
std::shared_ptr<Animal> animalPtr = std::make_shared<Animal>("動物", 4);
```

### **所有権の放棄**

所有権の放棄は、途中で行う事も可能。

```c++
//　メモリを解放する（所有権もリセットできる）
ptr.reset();
```

生ポインタを得る事も出来る。  
ただし推奨されないのは ```unique_ptr``` と同様。

```c++
// 生ポインタを取得する
int* temp1 = ptr.get();
```

### **shared_ptr の特徴**

```unique_ptr``` との大きな違いは所有権の「共有」が可能になる事。

```c++
// 複製により、所有権を共有できる
std::shared_ptr<int> temp = ptr;
```

「共有」しただけでは、生ポインタの時と同様に、  
各々が同じ所有権を持ててしまっている為、  
どこでメモリを解放するべきか分からない様になるが、  
```shared_ptr``` を利用して共有すると、  
同じ所有権を持っているオブジェクトがいなくなった時に、  
確保したメモリを自動的に解放する様になる。


```c++
#include <iostream>
#include <memory>

// メイン関数
int main()
{

	{
		// 所有権を持たない状態で宣言
		std::shared_ptr<int> ptr1;

		{
			// メモリを確保し、所有権を得る
			std::shared_ptr<int> ptr2 = std::make_shared<int>();
			*ptr2 = 100;

			// 複製により ptr2 が持つ所有権を ptr1 に共有する
			ptr1 = ptr2;
		}
		// ここで ptr2 の寿命が切れる
		// ptr2 が所有権を放棄するが、ptr1 が同じ所有権を共有している為、
		// ptr2 で確保したメモリ領域は解放されず残り続けており、
		// ptr1 を介して利用できる
		std::cout << *ptr1 << std::endl;
	}
	// ここで ptr1 の寿命が切れる
	// ptr2 で確保したメモリの所有権を持つオブジェクトがいなくなる為、
	// ここで漸くメモリの解放が行われる
	
	return 0;
}

```

### **shared_ptr の仕組み**

```shared_ptr``` は**参照カウント**と呼ばれる仕組みによって、  
所有権を持っているオブジェクトの数をカウントしている。  
所有権が共有される度にカウントが増え、放棄する度にカウントが減る。  
カウントがゼロになった時にそのメモリは必要なくなったと判断され、解放される。

## **スマートポインタの問題点**

```shared_ptr``` はある問題を持っている。  
次のコードを試してみてほしい。

```c++
#include <iostream>
#include <memory>

// クラス定義
class MyClass final
{
public:
	~MyClass() {
		std::cout << "デストラクタ" << std::endl;
	}

public:
	std::shared_ptr<MyClass> ptr_   = {};
	int						 value_	= 0;
};

// メイン関数
int main()
{
	{
		std::shared_ptr<MyClass> a = std::make_shared<MyClass>();
		std::shared_ptr<MyClass> b = std::make_shared<MyClass>();

		a->ptr_ = b;
		b->ptr_ = a;
	}
	
	return 0;
}

```

```shared_ptr``` で ```MyClass``` を二つインスタンス化、  
その所有権を持つオブジェクト a、b を、お互いのメンバ変数に複製する。  
この時、各オブジェクトが消えてもデストラクタは処理されなくなる。

お互いがお互いを ```shared_ptr``` で保持しているせいで、  
自分が消えても相手のメンバ変数に自分と同じ所有権を持つオブジェクトが残る。  
コーディング上では、a も b も寿命で破棄されているが、  
それぞれのインスタンスの所有権が相手に残っている為、メモリ解放が行われない。

こういった問題を**循環参照**と呼ぶ。  
メモリリークを起こす為、避けなければならない。

循環参照を避けるための方法として、  
生ポインタを利用するという方法があるが、  
それでは結局問題を起こしやすいコードに戻ってしまう。

---

## **weak_ptr**

```shared_ptr``` による循環参照を避ける方法として、  
```weak_ptr``` という物がある。

これは**所有権を持たずに利用するだけ**の場合に使用する。

```c++
#include <iostream>
#include <memory>

class MyClass final
{
public:
	~MyClass() {
		std::cout << "デストラクタ" << std::endl;
	}

public:
	std::weak_ptr<MyClass>  ptr_    = {};
	int						value_	= 0;

};

// メイン関数
int main()
{
	{
		std::shared_ptr<MyClass> p1 = std::make_shared<MyClass>();
		std::shared_ptr<MyClass> p2 = std::make_shared<MyClass>();

		p1->ptr_ = p2;
		p2->ptr_ = p1;
	}
	// 互いのメンバ変数が互いの所有権を持たない為、
	// p1 p2 が破棄されると正しくメモリ解放される
	
	return 0;
}

```

```weak_ptr``` を介してアクセスする場合、ひと手間必要になる。

```c++
// lock 関数を使い、shared_ptr オブジェクトを取得して漸くアクセス出来る
std::shared_ptr<MyClass> temp = p1->ptr_.lock();
temp->value_ = 1;
```

生ポインタを使うより手間ではあるが、  
生ポインタと違って「所有権を持っていない事」が一目でわかる為、  
間違えて `delete` してしまうといった事は防げる。
