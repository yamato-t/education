# **ステートパターン**

---

## **ステートパターンとは**

ステートパターンとは、オブジェクトの「状態」をクラスとして分離し、  
<span style="color: red;">**状態ごとに振る舞いを切り替える**</span>設計パターン。    
継承、多態性を活用した状態の切替を行う。 

ゲームキャラクターの行動制御（待機・移動・攻撃・ジャンプなど）に  
特によく使われる。  

---

## **なぜステートパターンが必要か**

### **if / switch による状態管理の問題**

状態管理を素直に実装すると、次のようなコードになりがち。  

```c++
void Player::Update()
{
    if (state == "idle") {
        // 待機中の処理
        if (input.IsPressedJump()) state = "jump";
        if (input.IsPressedMove()) state = "move";
    }
    else if (state == "move") {
        // 移動中の処理
        if (input.IsPressedJump()) state = "jump";
        if (input.IsReleasedMove()) state = "idle";
    }
    else if (state == "jump") {
        // ジャンプ中の処理
        if (isLanded) state = "idle";
    }
    // 状態が増えるほど分岐が膨れ上がる…
}
```

状態が増えるたびに `if` / `else if` が追加され、  
コードが肥大化・複雑化していく。  
一つの修正が別の状態に影響するバグも起きやすくなる。  

---

## **ステートパターンの仕組み**

### **三つの要素**

- <span style="color: red;">**State インターフェース（基底クラス）**</span>
    - 各状態に共通する関数を定義する抽象クラス。  
    - `Enter()`・`Update()`・`Exit()` を持つことが多い。  
- <span style="color: red;">**具体的な State クラス**</span>
    - 「待機」「移動」「ジャンプ」など、各状態をクラスとして実装。  
    - それぞれが State インターフェースを継承する。  
- <span style="color: red;">**Context クラス（状態を持つ側のクラス）**</span>
    - 現在の State インスタンスへのポインタを保持する。  
    - `Update()` の処理を現在の State に委譲する。  

`State` 毎の具体的な挙動は、派生クラスの `Enter()`・`Update()`・`Exit()` に実装する。  
派生クラスを `State` ポインタとして `Player` が保持し、関数を呼び出す事で、  
`Player` は具体的な内容を知らなくても、`state->各関数()` を呼ぶだけで良くなる。  

---

## **実装例**

### **State 基底クラス**

```c++
class Player; // 前方宣言

// プレイヤー状態基底
class PlayerState
{
public:
    virtual ~PlayerState() {}

    // 状態に入ったときに一度だけ呼ばれる
    virtual void Enter(Player& player) {}

    // 毎フレーム呼ばれる
    virtual void Update(Player& player) = 0;

    // 状態から出るときに一度だけ呼ばれる
    virtual void Exit(Player& player) {}
};
```

### **具体的な State クラス**

```c++
// 待機状態
class IdleState : public PlayerState
{
public:
    void Enter(Player& player) override {
        player.PlayAnimation("idle");
    }

    void Update(Player& player) override {
        if (input.IsPressedMove()) {
            player.ChangeState(new MoveState());
        }
        if (input.IsPressedJump()) {
            player.ChangeState(new JumpState());
        }
    }
};

// 移動状態
class MoveState : public PlayerState
{
public:
    void Enter(Player& player) override {
        player.PlayAnimation("move");
    }

    void Update(Player& player) override {
        player.Move();
        if (input.IsReleasedMove()) {
            player.ChangeState(new IdleState());
        }
    }
};
```

### **Context クラス（Player）**

```c++
class Player
{
public:
    Player() {
        currentState = new IdleState();
        currentState->Enter(*this);
    }

    ~Player() {
        delete currentState;
    }

    void Update() {
        // 現在の状態に処理を委譲するだけ
        currentState->Update(*this);
    }

    void ChangeState(PlayerState* newState) {
        currentState->Exit(*this);
        delete currentState;

        currentState = newState;
        currentState->Enter(*this);
    }

    void PlayAnimation(const std::string& name) { /* アニメーション再生 */ }
    void Move() { /* 移動処理 */ }

private:
    PlayerState* currentState;
};
```

### **Enter / Exit の役割**

`Enter()` と `Exit()` を用意することで、  
状態の切り替え時に一度だけ実行したい処理を書ける。  

| 関数 | タイミング | 使用例 |
|---|---|---|
| `Enter()` | 状態に入った瞬間 | アニメーション開始、初期化 |
| `Update()` | 毎フレーム | 移動・入力判定・遷移チェック |
| `Exit()` | 状態から出る瞬間 | エフェクトの停止、後処理 |

## **if 文による実装との比較**

| 比較項目 | if / switch 方式 | ステートパターン |
|---|---|---|
| 状態追加時の影響 | 既存の分岐に手を加える必要がある | 新しいクラスを追加するだけ |
| コードの見通し | 状態が増えると急速に悪化 | 状態ごとにファイルを分けられる |
| バグの局所化 | 全状態が一か所に混在 | 各状態が独立しているため追いやすい |
| 初期コスト | 少ない | クラス設計が必要でやや多め |

遷移する状態が少ない、小規模なケースでは `if` 文でも問題ない。  
多数になる、または今後増える見込みがある場合は  
ステートパターンへの移行を検討した方が良い。

## **注意点**

### **状態遷移が複雑になるケース**

状態数が多く、遷移条件も複雑な場合は  
<span style="color: red;">**状態遷移表（ステートマシン図）で整理してから**</span>作り始めた方が良い。  
コードだけで管理しようとすると、遷移の抜け漏れに気づきにくい。  

また昨今の各社エンジンでは、コーディングではなく  
**ノードベースでステート遷移を組み立てるツールや機能を用意している例が多い**。  
「ステート管理を行うためのツール」の実装を検討するのもあり。

### **メモリ管理に注意する**

上記の実装では `ChangeState()` のたびに `new` / `delete` が発生する。  
パフォーマンスが重要な場面では、  
あらかじめ全状態のインスタンスを生成しておき、切り替える方式も検討する。  

また、資料では簡潔にするために生ポインタを利用しているが、  
C++ っぽくスマートポインタでの実装をお勧めする。

## **要点**

| ポイント | 内容 |
|---|---|
| 目的 | 状態ごとの振る舞いをクラスに分離し、条件分岐を排除する |
| 構成要素 | State 基底クラス・具体 State・Context の三層 |
| 主な用途 | キャラクターの行動制御、UI の画面遷移、ゲームのフェーズ管理 |
| 導入タイミング | 状態が四つ以上、または今後増加が見込まれるとき |

ステートパターンの本質は、  
<span style="color: red;">**「状態に応じた if 文をなくし、状態そのものをオブジェクトにする」**</span>こと。  
一度身につけると、キャラクター制御だけでなく  
UI 管理やゲームシーン切り替えにも応用できる。
