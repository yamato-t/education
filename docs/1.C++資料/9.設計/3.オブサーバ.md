# **オブザーバーパターン**

---

## **オブザーバーパターンとは**

オブザーバーパターンとは、あるオブジェクトの状態が変化したとき、  
<span style="color: red;">**それを監視している複数のオブジェクトへ自動的に通知する**</span>設計パターン。  

「通知する側（Subject）」と「受け取る側（Observer）」を分離することで、  
**互いの実装を知らなくても連携できる構造**を作る。  
こちらも継承や多態性の活用によって実現するもの。

ゲーム開発では、スコア更新・HP変化・アイテム取得などの  
イベント通知に広く使われる。  

---

## **なぜオブザーバーパターンが必要か**

お互いが密接に結びついている状態を**密結合**と呼ぶ。  
これは設計において避けるべき状況であり、  
オブザーバーパターンはそれを回避するのに有用。

### **直接呼び出しによる密結合の問題**

スコアが変わったとき、UI・サウンド・実績システムへ  
それぞれ通知する場合を考える。  

```c++
// スコアを加算する
void Player::AddScore(int points) {
    score += points;

    // 関係するすべての相手を直接呼び出している
    ui->UpdateScoreDisplay(score);
    soundManager->PlaySound("score_up");
    achievementSystem->CheckScoreMilestone(score);
    // 通知先が増えるたびにここを修正しなければならない
}
```

この書き方では `Player` が UI・サウンド・実績など  
すべてのクラスへ直接依存している。  
通知先が一つ増える or 減るたびに `Player` のコードに追記する必要があり、  
変更の影響が広範囲に及ぶ。  

---

## **オブザーバーパターンの仕組み**

### **二つの要素**

- <span style="color: red;">**Subject（通知する側）**</span>
    - Observer を登録・解除する関数を持つ。  
    - 状態が変化したとき、登録済みの全 Observer に通知する。  
    - 通知先が何者かを知る必要がない。  
- <span style="color: red;">**Observer（通知を受ける側）**</span>
    - 通知を受け取る `OnNotify()` 関数を持つ。  
    - Subject に自分を登録することで通知を受け取れるようになる。  

### **構造の概要**

```
Subject（ScoreManager）
 ├─ AddObserver()     ← Observer の登録
 ├─ RemoveObserver()  ← Observer の解除
 └─ Notify()          ← 登録した Observer 全員に通知
        │
        ├──→ ScoreUI           （登録した Observer）
        ├──→ SoundManager      （登録した Observer）
        └──→ AchievementSystem （登録した Observer）
```

`Subject` は `Observer` のリストを持つだけで、  
中身が何のクラスかを意識させない。  
意識する必要も無い。

---

## **実装例**

### **Observer 基底クラス**

```c++
// Observer (通知を受ける側) の基底
class Observer
{
public:
    virtual ~Observer() {}

    // 通知を受け取ったときに呼ばれる
    // event にはイベントの種類、value には付随するデータを渡す
    virtual void OnNotify(const std::string& event, int value) = 0;
};
```

### **Subject クラス**

```c++
#include <vector>
#include <algorithm>

// Subject (通知を送る側) の基底
class Subject
{
public:
    // Observer 登録
    void AddObserver(Observer* observer) {
        observers.push_back(observer);
    }

    // Observer 解除
    void RemoveObserver(Observer* observer) {
        observers.erase(
            std::remove(observers.begin(), observers.end(), observer),
            observers.end()
        );
    }

protected:
    // 登録されている全 Observer に通知する
    void Notify(const std::string& event, int value) {
        for (Observer* observer : observers) {
            observer->OnNotify(event, value);
        }
    }

private:
    std::vector<Observer*> observers;
};
```

### **具体的な Subject クラス**

```c++
// スコアマネージャー（様々なオブジェクトに通知を送る側）
class ScoreManager : public Subject
{
public:
    // スコアを加算する
    void AddScore(int points) {
        score += points;

        // 直接呼び出しがなくなり、Notify 一本になる
        Notify("SCORE_CHANGED", score);
    }

    int GetScore() const { return score; }

private:
    int score = 0;
};
```

### **具体的な Observer クラス**

```c++
// スコア表示 UI（通知を受ける側）
class ScoreUI : public Observer
{
public:
    // 具体化した通知処理
    void OnNotify(const std::string& event, int value) override {
        if (event == "SCORE_CHANGED") {
            // UI の表示を更新する
            printf("Score: %d\n", value);
        }
    }
};

// サウンド再生（通知を受ける側）
class SoundManager : public Observer
{
public:
    // 具体化した通知処理
    void OnNotify(const std::string& event, int value) override {
        if (event == "SCORE_CHANGED") {
            PlaySound("score_up");
        }
    }
};
```

### **登録と使い方**

```c++
ScoreManager scoreManager;
ScoreUI      scoreUI;
SoundManager soundManager;

// Observer を登録する
scoreManager.AddObserver(&scoreUI);
scoreManager.AddObserver(&soundManager);

// スコアが増えると、登録済みの全 Observer に自動で通知される
scoreManager.AddScore(100);

// 不要になったら解除できる
scoreManager.RemoveObserver(&soundManager);
```

`ScoreManager` は `ScoreUI` や `SoundManager` を  
直接 `#include` する必要がなくなる。  
通知先を増やしたいときは新しい Observer を作って登録するだけで良い。  

通知先が増えても `ScoreManager` の中身を書き換える必要がなくなる。

---

## **直接呼び出しとの比較**

| 比較項目 | 直接呼び出し | オブザーバーパターン |
|---|---|---|
| 通知先追加時の修正 | Subject のコードを毎回変更 | Observer を追加して登録するだけ |
| 依存関係 | Subject が全通知先を知っている | Subject は Observer 基底クラスのみ知る |
| 通知先の着脱 | コードの変更が必要 | 実行時に `Add / Remove` で動的に変更可能 |
| 初期コスト | 少ない | 基底クラスの設計が必要でやや多め |

---

## **注意点**

### **解除忘れによる不正メモリアクセス**

<span style="color: red;">**Observer が破棄される前に必ず `RemoveObserver()` を呼ぶ。**</span>  
解除を忘れると、破棄済みのアドレスへ通知が飛び、  
クラッシュや未定義動作の原因になる。  

```c++
// デストラクタで必ず解除する
SoundManager::~SoundManager() {
    scoreManager->RemoveObserver(this);
}
```

なお delete したアドレスを保持し続けるポインタの事を**ダングリングポインタ**と呼ぶ。


### **通知の順序に依存しない設計にする**

Observer への通知順序は登録順になるが、  
その順序に依存した処理を書くと、登録順の変化でバグが生じる。  
各 Observer は他の Observer の処理結果に依存しない設計を保つ。  

---

## **要点**

| ポイント | 内容 |
|---|---|
| 目的 | 通知する側と受け取る側の依存関係を切り離す |
| 構成要素 | Subject（通知側）と Observer（受信側）の二層 |
| 主な用途 | スコア・HP・アイテム取得などのイベント通知 |
| 注意点 | Observer の解除忘れによるポインタ破壊に注意 |

オブザーバーパターンの本質は、  
<span style="color: red;">**「誰が聞いているかを知らなくても、変化を伝えられる仕組みを作る」**</span>こと。  
通知先が増減しても Subject のコードに触れなくて済む点が最大の利点。
