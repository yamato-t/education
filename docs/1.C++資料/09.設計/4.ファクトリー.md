# **ファクトリパターン**

---

## **ファクトリパターンとは**

ファクトリパターンとは、オブジェクトの生成処理を専用の場所に集約し、  
<span style="color: red;">**呼び出し側が生成するクラスの詳細を知らなくて済むようにする**</span>設計パターンの総称。  

「ファクトリ（工場）」という名の通り、  
何を作るかを指示するだけで、作り方はファクトリに任せる。  

ファクトリパターンには段階的に三つのバリエーションがある。  

| パターン名 | 概要 | 複雑さ |
|---|---|---|
| ファクトリ | 生成処理を一か所にまとめる | 低 |
| ファクトリメソッド | 生成処理をサブクラスに委ねる | 中 |
| アブストラクトファクトリ | 関連するオブジェクト群をまとめて生成する | 高 |

---

## **ファクトリ**

### **概要**

最も基本的な形。  
生成処理を一つのクラスに集約するだけのシンプルな仕組み。  

### **なぜ必要か**

敵キャラを生成する処理が各所に散らばっている場合を考える。  
ステージ毎に敵の生成処理を作成する。

```c++
// ステージ１で敵を生成する
void createEnemyInStage1() {
    Enemy* e1 = new Slime();
    Enemy* e2 = new Dragon();
    Enemy* e3 = new Slime();
}

// ステージ２で敵を生成する
void createEnemyInStage1() {
    Enemy* e1 = new Slime();
    Enemy* e2 = new Dragon();
    Enemy* e3 = new Slime();
    Enemy* e4 = new Skeleton();
}
```

この状態の時、仮に `Slime` のクラス名が変わってしまうと、  
`new Slime()` と記述しているすべての箇所を変更しなければならない。  
変更箇所が多く、漏れも発生しやすい。  

### **実装例**

```c++
class EnemyFactory
{
public:
    // 種類を渡すだけで適切な敵を生成して返す
    static Enemy* Create(std::string_view type)
    {
        if (type == "enemy1")  return new Slime();
        if (type == "enemy2") return new Dragon();
        if (type == "enemy3") return new Goblin();
        if (type == "enemy4") return new Skeleton();

        return nullptr;
    }
};

// 呼び出し側は type を渡すだけでよい
Enemy* enemy = EnemyFactory::Create("enemy1");
```

生成処理が `EnemyFactory` 一か所に集まるため、  
新しい敵を追加するときや、クラス名が変わった場合も修正箇所が明確になる。

## **ファクトリメソッド**

### **概要**

「どのオブジェクトを生成するか」の決定を派生クラスに委ねる。  
<span style="color: red;">**親クラスは生成処理の枠組みだけを定義し、具体的な生成は派生クラスが担う。**</span>  

### **なぜ必要か**

ステージの種類によって出現する敵が異なる場合を考える。  
ファクトリに分岐を増やし続けると肥大化していく。  

```c++
// ファクトリが膨れ上がった例
Enemy* Create(std::string_view stage, std::string_view type)
{
    if (stage == "forest")
    {
        if (type == "normal") return new Goblin();
        if (type == "boss")   return new ForestDragon();
    }
    else if (stage == "castle")
    {
        if (type == "normal") return new Knight();
        if (type == "boss")   return new DemonKing();
    }
    // ステージが増えるほど分岐が膨れ上がる…
}
```

大量のステージと敵種類が追加されてしまえば、  
いくら関数化していても分岐が多くて見にくい上に、間違えやすくなってしまう。


### **実装例**

```c++
// 親クラス：生成の枠組みだけを定義する
class Stage
{
public:
    virtual ~Stage() {}

    // 生成して初期化まで行う流れを定義
    Enemy* SpawnEnemy()
    {
        Enemy* enemy = CreateEnemy(); // 生成はサブクラスに任せる
        enemy->Initialize();
        return enemy;
    }

protected:
    // サブクラスが Override して具体的な生成を担う
    virtual Enemy* CreateEnemy() = 0;
};

// サブクラス：具体的な生成を実装する
class ForestStage : public Stage
{
protected:
    Enemy* CreateEnemy() override
    {
        return new Goblin();
    }
};

class CastleStage : public Stage
{
protected:
    Enemy* CreateEnemy() override
    {
        return new Knight();
    }
};
```

```c++
// 呼び出し側は Stage の種類だけを意識する
Stage* stage = new ForestStage();
Enemy* enemy = stage->SpawnEnemy();
```

新しいステージを追加するときは `Stage` を継承した  
新しいクラスを作るだけでよく、既存コードへの影響がない。

## **アブストラクトファクトリ**

### **概要**

「関連するオブジェクトの群（各部品）」をまとめて生成する。  
<span style="color: red;">**生成されるオブジェクト同士の組み合わせの一貫性を保証する**</span>のが目的。  

### **なぜ必要か**

UIのテーマ切り替えを考える。  
ダークテーマとライトテーマでは、ボタン・アイコン・背景が  
セットで切り替わらなければならない。  

```c++
// テーマの切り替えを個別に管理するとミスが起きやすい
Button*     btn  = new DarkButton();
Icon*       icon = new LightIcon();   // ← ライトとダークが混在してしまう
Background* bg   = new DarkBackground();
```

### **実装例**

```c++
// 生成する製品のインターフェース
class Button     { public: virtual void Draw() = 0; virtual ~Button() {} };
class Icon       { public: virtual void Draw() = 0; virtual ~Icon() {} };
class Background { public: virtual void Draw() = 0; virtual ~Background() {} };

// ファクトリのインターフェース：関連する製品群をまとめて定義する
class UIFactory
{
public:
    virtual ~UIFactory() {}
    virtual Button*     CreateButton()     = 0;
    virtual Icon*       CreateIcon()       = 0;
    virtual Background* CreateBackground() = 0;
};

// ダークテーマ用ファクトリ：必ずダーク系で揃う
class DarkUIFactory : public UIFactory
{
public:
    Button*     CreateButton()     override { return new DarkButton(); }
    Icon*       CreateIcon()       override { return new DarkIcon(); }
    Background* CreateBackground() override { return new DarkBackground(); }
};

// ライトテーマ用ファクトリ：必ずライト系で揃う
class LightUIFactory : public UIFactory
{
public:
    Button*     CreateButton()     override { return new LightButton(); }
    Icon*       CreateIcon()       override { return new LightIcon(); }
    Background* CreateBackground() override { return new LightBackground(); }
};
```

```c++
// 呼び出し側はファクトリを差し替えるだけでテーマ全体が切り替わる
UIFactory* factory = new DarkUIFactory();

Button*     btn = factory->CreateButton();
Icon*       ico = factory->CreateIcon();
Background* bg  = factory->CreateBackground();
// 必ずダーク系で統一される
```

## **三つのパターンの使い分け**

| パターン | 使うべき場面 | ゲームでの例 |
|---|---|---|
| ファクトリ | 生成の種類が少なく、増える見込みも少ない | 固定の敵リストから生成 |
| ファクトリメソッド | 生成するクラスの種類が今後増える見込みがある | ステージごとに異なる敵を生成 |
| アブストラクトファクトリ | 関連するオブジェクト群の組み合わせを保証したい | UIテーマ・プラットフォーム別アセット切り替え |

迷ったときは<span style="color: red;">**ファクトリから始め、拡張の必要が生じてから上位パターンへ移行する**</span>のが現実的。  
最初からアブストラクトファクトリを設計しようとすると、  
やりすぎな設計になりやすい。  

---

## **要点**

ファクトリパターンの本質は  
<span style="color: red;">**「`new` を呼ぶ責任を一か所に集め、呼び出し側から生成の詳細を隠す」**</span>こと。  
三つのバリエーションはその責任の持たせ方の違いにすぎず、  
目的はいずれも共通している。  

生成処理が各所に散らばり始めたと感じたとき、  
ファクトリパターンの導入を検討してみるといい。
