# **シングルトンパターン**

---

## **シングルトンパターンとは**

シングルトンパターンとは、あるクラスのインスタンスが  
<span style="color: red;">**プログラム全体で必ず一つしか存在しないことを保証する**</span>設計パターン。  

ゲーム開発では、サウンドマネージャーやスコア管理など、  
「複数存在すると困るもの」に頻繁に使われる。  

---

## **なぜシングルトンが必要か**

### **複数インスタンスが生まれる問題**

通常のクラスは `new` を呼ぶたびに新しいインスタンスが生成される。  

```c++
SoundManager* sm1 = new SoundManager();
SoundManager* sm2 = new SoundManager();
// sm1 と sm2 は別物。音量設定などがバラバラになる
```

サウンドマネージャーが二つ存在すると、  
片方で音量を変えても、もう片方には反映されない。  
こうした「状態の不一致」を防ぐためにシングルトンを使う。  

---

## **シングルトンの仕組み**

### **三つの要素**

シングルトンは以下の三点で成り立っている。  

- <span style="color: red;">**コンストラクタを `private` にする**</span>
    - これにより外部からインスタンスを生成できなくなる。  
    - クラス自身だけが生成を管理できる状態になる。  
- <span style="color: red;">**静的メンバ変数でインスタンスを保持する**</span>
    - クラス内部に自分自身のポインタを `static` で持つ。  
    - プログラム全体で一つのみ存在する領域に保存される。  
- <span style="color: red;">**静的関数 `GetInstance()` で唯一のインスタンスを返す**</span>
    - 初回呼び出し時だけ生成し、以降は同じインスタンスを返す。

コンストラクタを公開しない事で、  
第三者が好きにインスタンスを作成できなくなる。  
これによって**インスタンスが必ず一つしかない**状況を作る。  

### **基本的な実装**

仮にサウンドを管理するクラスをシングルトンで作る場合、  
下記のような形になる。

```c++
class SoundManager
{
public:
    // 唯一のインスタンスを返す静的関数
    static SoundManager* GetInstance() {
        if (instance == nullptr) {
            instance = new SoundManager();
        }
        return instance;
    }

    void PlaySound(const std::string& soundName) {
        // 再生処理
    }

    void SetVolume(float volume) {
        masterVolume = volume;
    }

private:
    // コンストラクタを private にして外部からの生成を禁止
    SoundManager() : masterVolume(1.0f) {}

    // 唯一のインスタンスを保持する静的変数
    static SoundManager* instance;

    float masterVolume;
};

// cpp ファイルで静的変数を初期化する
SoundManager* SoundManager::instance = nullptr;
```

### **使い方**

```c++
// どこからでも同じインスタンスにアクセスできる
SoundManager::GetInstance()->SetVolume(0.8f);
SoundManager::GetInstance()->PlaySound("bgm_title");
```

`new` を使わず `GetInstance()` 経由でアクセスするのが  
シングルトンの基本的な使い方になる。  

---

## **静的ローカル変数を使う方法**

### **Meyers' Singleton**

シングルトン実装には静的ローカル変数を使った書き方が推奨される。  
この方式は <span style="color: red;">**Meyers' Singleton**</span> と呼ばれる。  

```c++
class SoundManager
{
public:
    static SoundManager& GetInstance() {
        // 初回呼び出し時のみ生成される。以降は同じインスタンスを返す。
        static SoundManager instance;
        return instance;
    }

    void PlaySound(const std::string& soundName) { /* 再生処理 */ }
    void SetVolume(float volume) { masterVolume = volume; }

private:
    SoundManager() : masterVolume(1.0f) {}

    // コピーと代入を禁止する
    SoundManager(const SoundManager&) = delete;
    SoundManager& operator=(const SoundManager&) = delete;

    float masterVolume;
};
```

ポインタ版と比べた利点は以下の通り。  

| 比較項目 | ポインタ版 | Meyers' Singleton |
|---|---|---|
| `delete` の必要性 | 手動で必要 | 不要（自動で破棄） |
| スレッドセーフ | 非対応 | C++11以降は保証 |
| コピー禁止 | 別途対応が必要 | `= delete` で明示 |
| コード量 | 多め | 少なめ |

初学の段階では仕組みの理解にポインタ版を使い、  
実際の開発では Meyers' Singleton を採用すると良い。  

---

## **シングルトンの注意点**

### **乱用は設計を壊す**

シングルトンは便利だが、多用すると以下の問題が生じる。  

- <span style="color: red;">**グローバル変数と同じ問題が起こりがち**</span>
    - どこからでもアクセスできるため、依存関係が見えにくくなる。  
    - バグの原因がどこにあるか追いにくくなる。  
- <span style="color: red;">**影響範囲が大きくなりがち**</span>
    - とりあえずシングルトン化したあと、  
      どこからでも変更できる利便性から色々な処理を追加してしまい、  
      結果あらゆる影響を全体に与える大きなクラスになってしまう。  

### **適切な用途の目安**

シングルトンが向いているのは、以下のような場面。  

- ゲーム全体で一つだけ存在すべき管理クラス  
  （SoundManager、InputManager、ScoreManager など）  
- 設定ファイルや定数の保持  
- ログ出力クラス  

逆に、プレイヤーや敵キャラクターなど  
「複数存在し得るもの」にシングルトンを使うのは間違い。  
用途と特徴に合わせて適切に選ぶ必要がある。

---

## **要点**

| ポイント | 内容 |
|---|---|
| 目的 | インスタンスを一つに制限する |
| 手段 | コンストラクタの `private` 化 ＋ 静的関数によるアクセス |
| 推奨実装 | Meyers' Singleton |
| 注意点 | 乱用せず、本当に一つでいい場面だけに使う |

シングルトンはデザインパターンの入門として  
最もシンプルな部類に入るが、  
「なぜ一つである必要があるのか」を意識して使うことが重要。
