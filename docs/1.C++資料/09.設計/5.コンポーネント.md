# **コンポーネントパターン**

---

## **コンポーネントパターンとは**

コンポーネントパターンとは、オブジェクトの機能を独立した小さなクラス  
（コンポーネント）に分割し、それらを組み合わせて一つのオブジェクトを  
構成する設計パターン。  

Unity の `AddComponent` や UE5 の `UActorComponent` がまさにこの考え方。  
<span style="color: red;">**「継承より合成」**</span>という設計原則を体現したパターン。  

---

## **なぜコンポーネントパターンが必要か**

### **深い継承階層の問題**

ゲームオブジェクトを継承で表現しようとすると、  
次のような階層になりがち。  

```
GameObject             (描画)
 ├─ Character          (移動・HP・攻撃)
 │   ├─ Player        （入力）
 │   └─ Enemy         （AI）
 │
 └─ Prop               
     ├─ StaticProp    （描画のみ）
     └─ MovableProp   （移動）
```

ここに、下記を満たした罠オブジェクトを追加する場合を考える。

- 描画する
- 移動する
- 攻撃する
- AI はない
- HP はない
- 入力は受けない

素直に考えると `Character` が一番近い。  
その為 `Character` を継承した派生クラスとして作成する。  
だがこの時、**不必要な機能まで継承で取り込んでしまう事になる**

```c++
// 罠オブジェクト
class TrapObject : public Character 
{
    // HP は不要なのに機能を継承してしまう
};
```

これだけであればまだ単純だが、  
実際のゲーム制作では多種多様な機能を持ったオブジェクトの実装が必要になる。

その実装で継承を重ねるほどクラスは肥大化し、  
大量の不要な機能まで引き継いだ**神クラス**が生まれやすくなる。  

---

## **コンポーネントパターンの仕組み**

### **二つの要素**

- <span style="color: red;">**Component（コンポーネント）**</span>
    - 「移動」「描画」「入力」「HP」など、機能単位のクラス。  
    - それぞれが独立しており、互いを知らなくてよい。  
- <span style="color: red;">**GameObject（コンテナ）**</span>
    - コンポーネントを保持するだけの入れ物。  
    - 機能の実装は持たず、各コンポーネントに処理を委譲する。  

### **構造の概要**

```
GameObject
 ├─ TransformComponent  （位置・回転・スケール）
 ├─ RenderComponent     （描画）
 ├─ InputComponent      （入力）
 └─ HealthComponent     （HP管理）
```

必要なコンポーネントだけを組み合わせることで、  
どんな性質のオブジェクトでも柔軟に表現できる。  

---

## **実装例**

### **Component 基底クラス**

```c++
class GameObject; // 前方宣言

class Component
{
public:
    virtual ~Component() {}
    virtual void Update(GameObject& owner) {}
    virtual void Render(GameObject& owner) {}
};
```

### **具体的なコンポーネント**

```c++
// 移動を担うコンポーネント
class TransformComponent : public Component
{
public:
    float x = 0.0f, y = 0.0f;

    void Move(float dx, float dy)
    {
        x += dx;
        y += dy;
    }
};

// 描画を担うコンポーネント
class RenderComponent : public Component
{
public:
    void Render(GameObject& owner) override
    {
        // スプライト描画処理
    }
};

// HP 管理を担うコンポーネント
class HealthComponent : public Component
{
public:
    HealthComponent(int maxHp) : hp(maxHp), maxHp(maxHp) {}

    void TakeDamage(int damage)
    {
        hp = std::max(0, hp - damage);
    }

    bool IsDead() const { return hp <= 0; }

private:
    int hp;
    int maxHp;
};
```

### **GameObject クラス**

コンポーネントを文字列をキーとしたマップで管理する。  
`AddComponent()` で登録し、`GetComponent()` で取り出す。  

```c++
#include <string>
#include <map>

class GameObject
{
public:
    // 名前を指定してコンポーネントを登録する
    void AddComponent(const std::string& name, Component* component)
    {
        components[name] = component;
    }

    // 名前を指定してコンポーネントを取得する
    Component* GetComponent(const std::string& name)
    {
        auto it = components.find(name);
        if (it != components.end())
        {
            return it->second;
        }
        return nullptr;
    }

    void Update()
    {
        for (auto& pair : components)
        {
            pair.second->Update(*this);
        }
    }

    void Render()
    {
        for (auto& pair : components)
        {
            pair.second->Render(*this);
        }
    }

private:
    std::map<std::string, Component*> components;
};
```

### **組み合わせて使う**

```c++
// プレイヤー：移動・描画・HP すべてを持つ
GameObject player;
player.AddComponent("transform", new TransformComponent());
player.AddComponent("render",    new RenderComponent());
player.AddComponent("health",    new HealthComponent(100));

// 背景オブジェクト：描画だけを持つ
GameObject background;
background.AddComponent("render", new RenderComponent());

// 罠オブジェクト：移動と描画だけを持つ（HP なし）
GameObject trap;
trap.AddComponent("transform", new TransformComponent());
trap.AddComponent("render",    new RenderComponent());

// コンポーネントへのアクセス
// GetComponent() の戻り値は Component* なのでキャストが必要
HealthComponent* health =
    static_cast<HealthComponent*>(player.GetComponent("health"));

if (health != nullptr)
{
    health->TakeDamage(30);
}
```

継承では表現しにくかった「移動するが HP を持たない」オブジェクトも、  
コンポーネントの組み合わせで自然に表現できる。

## **継承との比較**

| 比較項目 | 継承による設計 | コンポーネントパターン |
|---|---|---|
| 機能の追加 | 基底クラスの修正、または再継承が必要 | コンポーネントを追加するだけ |
| 不要な機能 | 基底クラスから引き継いでしまう | 必要なものだけを選んで持てる |
| 新種オブジェクト | 階層のどこに置くか悩む | 組み合わせで自由に表現できる |
| 実行時の変化 | 継承構造は静的で変えられない | 実行時にコンポーネントの着脱が可能 |

## **注意点**

### **コンポーネント間の依存に注意**

コンポーネントは互いに独立しているのが理想だが、  
実際には依存してしまう事がある。  

```c++
void Update(GameObject& owner) override
{
    // 別のコンポーネントを owner 経由で取得するのは許容範囲
    TransformComponent* transform =
        static_cast<TransformComponent*>(owner.GetComponent("transform"));

    if (transform != nullptr) { /* transform を使った処理 */ }
}
```

`GetComponent()` 経由での参照は許容範囲だが、  
コンポーネント同士が直接ポインタを持ち合う設計は避ける。  

### **コンポーネント数が増えすぎる場合**

機能をすべてコンポーネントに分割しようとすると、  
逆に管理が煩雑になることもある。  
「一つの責務として切り出す意味があるか」を基準にして  
粒度を判断すると良い。  

### **コンポーネントの登録方法**

サンプルでは分かりやすさ重視の為、  
文字列をキーにコンポーネントを管理しているが、  
実際の実装では、クラスを表す ID 等をキーとして管理した方が適切。

## **まとめ**

| ポイント | 内容 |
|---|---|
| 目的 | 機能を独立したコンポーネントに分割し、組み合わせで表現する |
| 解決する問題 | 深い継承階層・神クラスの肥大化 |
| 主な用途 | ゲームオブジェクトの機能管理（Unity・UE5 の設計思想） |
| 設計原則 | 継承より合成（Composition over Inheritance） |

コンポーネントパターンの本質は、  
<span style="color: red;">**「何者であるか（is-a）ではなく、何を持つか（has-a）で設計する」**</span>こと。  
継承で表現しようとして行き詰まったとき、  
特に「必要ない機能まで大量に継承してしまいそう」となった時、  
コンポーネントへの切り替えを検討してみるといい。
