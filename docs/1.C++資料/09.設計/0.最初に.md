# 設計

---
## **そもそも設計とは**

設計とは**全体を完成させるためにどの様な部品が必要で、**  
**それらをどの様に組み合わせていくか**を整理する工程を指す。  
細かく分けると「外部設計」「内部設計」「プログラム設計」など色々あるが、  
ひとまずそれらの分類はここでは割愛する。

設計する目的は<span style="color: red;">**完成させる為に必要な物と処理と流れを明確にする為**</span>。  
設計が終わった結果はクラス図やフローチャートといった、目に見える形にするのがいい。  
それらを見ながらコーディングすれば、  
自然と目的の物を完成させる事ができる、という事になる。

その為、設計は**コーディングする前**に行う。

基本的に何かを作るという事は、それに必要な物や処理を考えなければできない。  
今まで出していた課題程度であっても、  
それぞれ頭の中で「あれが必要で」「こんなことを行わないといけない」  
といった検討を繰り返していたと思う。  
それも一種の設計といえる。  

---
## **設計の良し悪し**
良い設計の例には、下記のような内容がある

 - 変更（拡張）しやすい  
 - 一部に発生した問題がほかに影響を与えない
 - 再利用が簡単

「何が良いのか」（または「何が悪いのか」）をはっきりさせ、  
その逆を考えれば、良し悪しは判断しやすい。

ほかにも色々あると思うがまずはこれらが基本になる。

## **設計する上でまず考える事**

まず全体の作りたい物をハッキリさせ、その上で下記を考えてみる。

- <span style="color: red;">**部品に分解する**</span> 
- <span style="color: red;">**データとそれを扱う処理を分ける**</span>
- <span style="color: red;">**部品同士の依存関係を減らす**</span>

### **部品に分解する**

例えば車に着目した場合、  
```車全体を一つの部品``` とする事もできれば、  
```エンジンやアクセル、ブレーキをそれぞれ部品とし、車はそれらの集合``` とする事もできる。

- **車全体を一つの部品**とした場合  
燃料を動力に変え、加速と減速ができ、物を乗せ、ＧＰＳによるナビを行い…等、  
<span style="color: red;">**一つの部品として行うべき働きがとにかく多い**</span>。

- **エンジン、アクセル、ブレーキをそれぞれ部品とし、車はそれらの集合とした場合**  
   　・エンジンが行う働き：送られてきた燃料を動力に変える  
   　・アクセルが行う働き：踏まれ具合によって加速する  
   　・ブレーキが行う働き：踏まれ具合によって減速する  
といった様に、行うべき働きが限定され、  
<span style="color: red;">**各部品が持つ責任を明確していく事が出来る**</span>。  

車全体が一つの部品になっていると、問題が出た際にどこに問題があるのか探しにくい。  
それぞれを部品としていれば、問題が出た際に、  
その問題の内容によってどの部品の問題か判断しやすく、修正も行いやすい。

「**この働きはこの部品が責任をもって行っている**」という事が、  
明確であればあるほど、問題の対処や変更が行いやすくなってくる。

### **データとそれを扱う処理を分ける**
日本語で働きを説明した際の、
「名詞」と「動詞」に注目してみるといい。

- ```エンジン```の場合  
「送られてきた燃料」が名詞、「動力に変える」が動詞になる。
- ```アクセル```の場合  
「踏まれ具合」が名詞、「加速する」が動詞になる。

上記の名詞はデータ、動詞は処理に置き換えられる。

**データと処理を一緒に扱う一つの部品として分解した場合**と、  
**データを受け取って処理を行う部品として分解した場合**では、  
設計結果が大きく変わってくる。

例えば```アクセル```は「踏まれ具合」によって「加速する」様になっているが、  
**データと処理を一緒に扱う部品として分解する**と、  
<span style="color: red;">**バイクでは再利用出来ない事**</span>がわかるだろうか。  
バイクは「踏まれ具合」ではなく「捻り具合」によって加速する為。  
```アクセル```という部品の再利用性が、どの様に分解するかで変わってくる。

### **部品同士の依存関係を減らす**

```エンジンやアクセル、ブレーキ、ハンドルをそれぞれ部品とし、車はそれらの集合``` とした場合、  
当然```エンジン```や```アクセル```や```ブレーキ```や```ハンドル```が働かなければ```車```は動かない。  
ただ各部品は、独立して働けるようにしておいた方が好ましい。

ブレーキ踏まなくてもハンドルは回る様になっててほしいし、  
アクセル無くてもブレーキは利くようになっていてほしい。

仮に```ハンドル```がないと```ブレーキ```が働かないといった依存関係にした場合、  
```ハンドル```に何らかの変更を行う事で```ブレーキ```が正しく働かなくなる可能性が出てくる。  
<span style="color: red;">**一部の変更が意外な所に影響を与えてしまい、大きな問題に発展する危険がある**</span>。

これは責任の話にも似通ってくる  
ハンドルはあくまで「進行方向」の責任を担っており、ブレーキは「減速」の責任を担っている。  
「```ハンドル```がないと```ブレーキ```が働かないといった依存関係」がある場合、  
```ハンドル```が「減速」の責任まで担ってしまっている事になる。  

つまりハンドルが壊れるとブレーキも壊れる事になる。  
どこかが壊れると、**連鎖的にいろんな所が壊れる様な作りはとにかく避けた方が良いのは間違いない**。  
これはプログラムにもいえる事。

## **プログラム寄りの話**

### **日本語で正しく説明出来る継承関係を作る**
設計した内容は最終的にプログラム言語を使ってコーディングされる物なので、  
日本語で説明した際に不自然だったり矛盾していたりすると、必ず変な結果になってる。  

設計において継承は非常に重要。  
何をどう継承させるかというのは設計の根幹に関わってくる。

ここで気にしてもらいたい事として、**継承**は  
<span style="color: red;">**「○○はxxです」「○○はxxできます」という日本語が成り立つ関係でのみ行う**</span>   
という事が挙げられる。  

#### **「○○はxxです」の関係**

まずは **「○○はxxです」** の例を挙げる。  

- 「猫」クラスが「動物」クラスを継承する　　→　「猫は動物です」 OK  
- 「犬」クラスが「生物」クラスを継承する　　→　「犬は生物です」 OK  
- 「鳥」クラスが「羽」クラスを継承する　　　→　「鳥は羽です」 ?

最後だけおかしい。**鳥=羽ではない**。  
この場合「鳥は羽を持っている」という日本語が正しい。  
<span style="color: red;">メンバ変数に羽クラスのインスタンスを持っている方が意味として通じやすい</span>。

「○○はxxです」が成り立つ関係は <span style="color: red;">**is-aの関係**</span>と呼ばれ、   
「○○はxxを持っている」が成り立つ関係は <span style="color: red;">**has-aの関係**</span>と呼ばれる。  
継承するのか、メンバに持つのかを判断する指針としてもらいたい。


#### **「○○はxxできます」の関係**

同じように考えてみる。  

- 「人」クラスが「走る」クラスを継承する　→　「人は走れます」 OK  
- 「車」クラスが「走る」クラスを継承する　→　「車は走れます」 OK  
- 「木」クラスが「走る」クラスを継承する　→　「木は走れます」 ?

基底クラスは全て「走る」という動作を表しているだけであり、  

- 「人」なら「足を動かして走る」  
- 「車」なら「四つのタイヤを回転させて走る」  

といった様に、  
<span style="color: red;">**派生先で動作の内容を具体的にオーバーライドさせる事が必須となってくる**</span>。  

ただ最後だけおかしい。木は走れない。  
このような場合は継承するべきではない。

「○○はxxできます」が成り立つ関係は、<span style="color: red;">****can-doの関係**</span>と呼ばれる。  
基底クラスがインターフェイスクラスの場合、この関係が成り立つかどうかが指針となる。


### **デザインパターンを利用する**
プログラムにおいて、ありがちな問題とその対処法まとめた、  
<span style="color: red;">**デザインパターン**</span>と呼ばれるプログラムの設計パターンがある。  

一般的に知られているものも多く、自身の実装を他人に説明する際に、  
「ここは○○パターンで設計してます」といった様に説明する事もできる。  
自身の作品に積極的に取り入れて「ここは○○パターン使いました」と言えるのが好ましい。

主に下記の内容に関する設計となっている。

- インスタンスの生成
- インスタンスの構造
- インスタンスの振る舞い（処理）

設計内容は具体的に提示されているが、  
実装自体が具体的に提示さえている訳ではない。  
設計パターンを把握し、自身の実装に上手く取り入れていく必要がある。
