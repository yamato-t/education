# **concept**

---
concept は**テンプレートの実引数の型が、指定の要件を満たせるかどうかを判定する為の仕組み**。  

C++ 20 から登場した仕組みであり、  
これまで難度の高かったテンプレートの実装を簡易に実現させる事が出来るようになった。

下記サンプルコードになる。

```c++
include <iostream>

// draw 関数を持っているクラス
class HasDrawA {
public:
    void draw() { std::cout << "HasDrawA::draw() が呼び出されました" << std::endl; }
};

// draw 関数を持っているクラス
class HasDrawB {
public:
    void draw() { std::cout << "HasDrawB::draw() が呼び出されました" << std::endl; }
};

// draw 関数を持っていないクラス
class NoneDraw {};

// T が draw という関数を持っているかどうかを判断する為のコンセプト
template <class T>
concept HasDrawObject = requires(T& t) {
    t.draw();  // T 型で draw という関数が呼び出せる、という意味になる
};

// draw 関数を呼び出すためのテンプレート関数
// テンプレート仮引数にコンセプトを指定する
template <HasDrawObject T>
void callDraw(T& x) {
    x.draw();
}

// エントリー関数
int main() {
    HasDrawA a;
    HasDrawB b;
    NoneDraw n;

    callDraw(a); // HasDrawA は draw を呼び出せるので問題ない
    callDraw(b); // HasDrawB は draw を呼び出せるので問題ない
    callDraw(n); // NoneDraw は draw を呼び出せないのでコンパイルエラー

    return 0;
}
```

具体的には下記個所が **Concept** と呼ばれる部分になる。

```C++
// T が draw という関数を持っているかどうかを判断する為のコンセプト
template <class T>
concept HasDrawObject = requires(T& t) {
    t.draw();  // T 型で draw という関数が呼び出せる、という意味になる
};
```

**T 型のオブジェクトが draw という関数を呼び出せる**というルールを表している。  
そのルールに `HasDrawObject` という名前を付けているという認識で問題ない。

そしてそのルールを下記のテンプレート関数で利用している。

```c++
// draw 関数を呼び出すためのテンプレート関数
// テンプレート仮引数にコンセプトを指定する
template <HasDrawObject T>
void callDraw(T& x) {
    x.draw();
}
```

テンプレート仮引数に注目する。  
今までは `<class T>` といった記述が大半だったが、`<HasDrawObject T>`に変化している。  
これによりテンプレート関数 `callDraw` は、  
**引数の型が HasDrawObject というルールを満たしていないと呼び出せない**  
という事になる。

その為、main 関数内の下記でコンパイルエラーが出る。  
```C++
callDraw(n); // NoneDraw は draw を呼び出せないのでコンパイルエラー
```
理由は n の型である `NoneDraw` が **draw 関数を持っていない為**。

## **Concept と constexpr if によるエラー回避**

上記 Concept を利用したコードだが、  
Concept を利用せずともコンパイルエラーを出す事は可能。  
単純にルールを作るだけなら今までのテンプレートでも簡単に実装出来る。

Concept を利用する事で得られる大きな恩恵は、  
**ルールに沿わない場合もコンパイルエラーを出さずに例外として処理を分岐出来る**  
という点にある。

例えば上記のコードで  
<strong> draw 関数を持たない場合は draw を呼び出さず無視させたい</strong>  
とする。

本来、上記を実現させるには非常に難解なコーディングが必要になる。  
ところが Concept と 前述の **constexpr if** を組み合わせる事で簡単に実現する。  
テンプレート関数を下記のように変更するのみ。

```c++
// draw 関数を呼び出すためのテンプレート関数
template <class T>
void callDraw(T& x) {
    if constexpr (HasDrawObject<T>) {
        x.draw();    
	} else {
        std::cout << "draw() が呼び出せません" << std::endl;
	}
}
```

変更点は以下の通り

* テンプレート仮引数を `<class T>` に変更して `calldraw` をどの様な型でも受け入れる関数にする
* `constexpr if` で T が `HasDrawObject` のルールを満たすかどうか判定する
* ルールを満たす場合だけ `draw` 関数を呼び出す

こうすることで、実引数の型がルールを満たす場合は draw 関数が呼び出され、  
満たされない場合は else の処理が行われるようになり、
コンパイルエラーは発生しなくなる。

なおこの場合は `constexpr` を削除してしまうとコンパイルエラーが出る。  
なぜなら、通常の if  文では  
**T が draw 関数を持っていなくても、if 文の中身の処理が存在している為**。  
つまり T が draw 関数を持っていなくても、  
`x.draw()`を呼び出すコードがテンプレート実体化時にコンパイルされてしまう。

`constexpr if` の場合、条件を満たさない分岐の処理は**無効になる（存在しない）**と前述した。  
T がルールを満たさない型だった場合、ルールを満たしている部分のコードが無効になる為、  
**本来コンパイルエラーが出ていた個所が無視されるようになり、結果コンパイルエラー自体が発生しなくなる**。

これにより、テンプレート利用時に処理の融通が効きやすく、  
コンパイルエラーを出さずに例外か否かを分かりやすくさせる事が可能になった。
