# **スレッドの危険性**

---

## **データ競合**

<span style ="color: red;">**データ競合**</span> とは、  
複数のスレッドが同じメモリ領域に対して競い合うように  
読み書きを行う事で、結果が不定になる問題を指す。

「競合」という名の通り、どのスレッドが先に実行されるかによって  
毎回異なる結果になったり、値が壊れたりする。

こういった問題を起こすコードの区間を  
<span style ="color: red;">**クリティカルセクション**</span> という。

例えばグローバル変数への読み書きがある箇所がこれに当たる。  
クリティカルセクションを適切に保護しなければ、  
スレッドが増えるほど深刻なバグが発生しやすくなる。

### **データ競合が起きるコード例**

下記は複数のスレッドから共有カウンタをインクリメントする例。

```c++
#include <iostream>
#include <thread>

// 複数スレッドから共有される変数
int g_counter = 0;

void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        // データ競合が起きる箇所
        ++g_counter;
    }
}

int main()
{
    std::thread tA(Increment);
    std::thread tB(Increment);

    tA.join();
    tB.join();

    // 期待値は 200000 だが、実際には毎回異なる値になる
    std::cout << "カウンタ: " << g_counter << std::endl;

    return 0;
}
```

二つのスレッドが合わせて 200000 回インクリメントするので、  
結果は 200000 になりそうに見える。  
しかし実際に実行すると 200000 より小さい値になり、  
しかも実行するたびに値が変わる。

### **なぜ値が壊れるのか**

`++g_counter` は一行で書けるが、CPU が実際に行う処理は  
以下の三つの手順に分かれている。

```
1. g_counter の値をレジスタに読み込む
2. レジスタの値に 1 を加算する
3. レジスタの値を g_counter に書き戻す
```

スレッドAとスレッドBが同時にこの処理を行うと、  
次のような事態が起きる可能性がある。

```
スレッドA: g_counter を読み込む（値は 5）
スレッドB: g_counter を読み込む（値は 5）
スレッドA: 5 + 1 = 6 を書き戻す
スレッドB: 5 + 1 = 6 を書き戻す（Aの書き込みが上書きされる）
```

本来なら 7 になるべき所が 6 になってしまう。  
これが積み重なっていく為、最終的な値は大きくズレていく。

C++ の標準では、同期なしに複数のスレッドが同じメモリ領域にアクセスし、  
そのうち少なくとも一方が書き込みを行う事を<span style ="color: red;">**未定義動作**</span> としている。  
「書込同士」だけでなく「書込と読込」の組み合わせも同様となる。  
未定義動作はコンパイラが予測不能な最適化を行う余地を与える為、  
「たまたま正しく動く」状態ですら安全とはいえない。

## **デッドロック**

<span style ="color: red;">**デッドロック**</span> とは、  
複数のスレッドが互いに相手のリソース解放を待ち続け、  
どちらも先に進めなくなる状態を指す。  
プログラムがフリーズしたように見えるが、エラーは発生しない為  
原因に気づきにくい。

### **デッドロックが起きるコード例**

二つのスレッドが二つのミューテックスを逆順にロックしようとする例。

```c++
#include <iostream>
#include <thread>
#include <mutex>

std::mutex g_mutexA;
std::mutex g_mutexB;

void ThreadA()
{
    g_mutexA.lock();                                     // 1. A をロック
    std::cout << "スレッドA: mutexA をロック" << std::endl;

    g_mutexB.lock();                                     // 3. B を待機し続ける
    std::cout << "スレッドA: mutexB をロック" << std::endl;

    g_mutexB.unlock();
    g_mutexA.unlock();
}

void ThreadB()
{
    g_mutexB.lock();                                     // 2. B をロック
    std::cout << "スレッドB: mutexB をロック" << std::endl;

    g_mutexA.lock();                                     // 4. A を待機し続ける
    std::cout << "スレッドB: mutexA をロック" << std::endl;

    g_mutexA.unlock();
    g_mutexB.unlock();
}

int main()
{
    std::thread tA(ThreadA);
    std::thread tB(ThreadB);

    tA.join(); // 永遠に終わらない
    tB.join();

    return 0;
}
```

処理の流れを追うと次のようになる。

```
スレッドA: mutexA をロックする
スレッドB: mutexB をロックする
スレッドA: mutexB をロックしようとする → スレッドBが持っている為、待機
スレッドB: mutexA をロックしようとする → スレッドAが持っている為、待機
```

スレッドAはBが `mutexB` を解放するのを待ち、  
スレッドBはAが `mutexA` を解放するのを待っている。  
しかし両者とも待機中なので解放は永遠に起きない。  
これがデッドロックとなる。

### **デッドロックを避けるには**

最も基本的な対策は、  
<span style ="color: red;">**複数のミューテックスは常に同じ順番でロックする**</span> こと。  
上記の例では、スレッドBも `mutexA → mutexB` の順でロックすれば  
デッドロックは発生しない。
