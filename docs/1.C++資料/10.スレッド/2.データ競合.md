# **危険性その① データ競合**

---

## **データ競合とは**

<span style ="color: red;">**データ競合**</span> とは、  
複数のスレッドが同じメモリ領域に対して競い合うように  
読み書きを行う事で、結果が不定になる問題を指す。

「競合」という名の通り、どのスレッドが先に実行されるかによって  
毎回異なる結果になったり、値が壊れたりする。

こういった問題を起こすコードの区間を  
<span style ="color: red;">**クリティカルセクション**</span> という。  
例えばグローバル変数への読み書きがある箇所がこれに当たる。  
クリティカルセクションを適切に保護しなければ、  
スレッドが増えるほど深刻なバグが発生しやすくなる。

### **データ競合が起きるコード例**

下記は複数のスレッドから共有カウンタをインクリメントする例。

```c++
#include <iostream>
#include <thread>

// 複数スレッドから共有される変数
int g_counter = 0;

void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        // データ競合が起きる箇所
        ++g_counter;
    }
}

int main()
{
    std::thread tA(Increment);
    std::thread tB(Increment);

    tA.join();
    tB.join();

    // 期待値は 200000 だが、実際には毎回異なる値になる
    std::cout << "カウンタ: " << g_counter << std::endl;

    return 0;
}
```

二つのスレッドが合わせて 200000 回インクリメントするので、  
結果は 200000 になりそうに見える。  
しかし実際に実行すると 200000 より小さい値になり、  
しかも実行するたびに値が変わる。

### **なぜ値が壊れるのか**

`++g_counter` は一行で書けるが、CPU が実際に行う処理は  
以下の三つの手順に分かれている。

```
1. g_counter の値をレジスタに読み込む
2. レジスタの値に 1 を加算する
3. レジスタの値を g_counter に書き戻す
```

スレッドAとスレッドBが同時にこの処理を行うと、  
次のような事態が起きる可能性がある。

```
スレッドA: g_counter を読み込む（値は 5）
スレッドB: g_counter を読み込む（値は 5）
スレッドA: 5 + 1 = 6 を書き戻す
スレッドB: 5 + 1 = 6 を書き戻す（Aの書き込みが上書きされる）
```

本来なら 7 になるべき所が 6 になってしまう。  
これが積み重なっていく為、最終的な値は大きくズレていく。

C++ の標準では、同期なしに複数のスレッドが同じメモリ領域にアクセスし、  
そのうち少なくとも一方が書き込みを行う事を  
<span style ="color: red;">**未定義動作**</span> としている。  
「書込同士」だけでなく「書込と読込」の組み合わせも同様となる。  
未定義動作はコンパイラが予測不能な最適化を行う余地を与える為、  
「たまたま正しく動く」状態ですら安全とはいえない。

---

## **回避方法**

クリティカルセクションの安全性を確保する手段は多岐にわたり、  
スピンロック、セマフォ、読み書きロック、ロックフリーアルゴリズムなど  
様々な方法が存在する。  
ここではその入口として、C++ 標準ライブラリが提供する  
`std::mutex` と `std::atomic` を取り上げる。

### **ミューテックス**

<span style ="color: red;">**ミューテックス（Mutex）**</span> とは、  
Mutual Exclusion（相互排他）の略で、  
同時に一つのスレッドしかロックを取得できない仕組みを指す。

`<mutex>` ヘッダに含まれる `std::mutex` を使う。

```c++
#include <iostream>
#include <thread>
#include <mutex>

int g_counter = 0;
std::mutex g_mutex; // ミューテックスを用意する

void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        g_mutex.lock();   // ロックを取得（他スレッドは待機）
        ++g_counter;      // クリティカルセクション
        g_mutex.unlock(); // ロックを解放
    }
}

int main()
{
    std::thread tA(Increment);
    std::thread tB(Increment);

    tA.join();
    tB.join();

    // 正しく 200000 になる
    std::cout << "カウンタ: " << g_counter << std::endl;

    return 0;
}
```

`lock()` を呼んだスレッドだけがクリティカルセクションに入れる。  
別のスレッドが `lock()` を呼ぶと、先のスレッドが `unlock()` するまで待機する。

### **lock_guard**

`lock()` と `unlock()` を手動で呼ぶと、  
例外が発生した場合や途中で `return` した場合に  
`unlock()` を呼び忘れるリスクがある。  
`unlock()` が呼ばれないと他のスレッドが永遠に待ち続けるデッドロックが起きる。

この問題を防ぐ為に <span style ="color: red;">**std::lock_guard**</span> を使う。  
スコープを抜ける際に自動でロックを解放してくれる。

```c++
void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        // スコープを抜けると自動的に unlock される
        std::lock_guard<std::mutex> lock(g_mutex);
        ++g_counter;
    }
}
```

`std::lock_guard` は RAII（Resource Acquisition Is Initialization）  
という手法を使っており、構築時にロックを取得し、  
破棄時に自動でロックを解放する。  
手動で `unlock()` を呼ぶ必要がなくなる為、安全性が高い。

### **アトミック操作**

ミューテックスは強力だが、ロックの取得・解放にコストがかかる。  
単純な数値の読み書きのみが目的であれば、  
<span style ="color: red;">**std::atomic**</span> を使う方が効率的となる。

```c++
#include <iostream>
#include <thread>
#include <atomic>

// int の代わりに atomic<int> を使う
std::atomic<int> g_counter(0);

void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        ++g_counter; // アトミック操作で安全にインクリメント
    }
}
```

`std::atomic<T>` は、指定した型 `T` に対する読み書きを  
分割不可能な一つの操作（アトミック操作）として実行する事を保証する。  
ミューテックスよりも低コストで動作する為、  
単純なカウンタやフラグに向いている。
