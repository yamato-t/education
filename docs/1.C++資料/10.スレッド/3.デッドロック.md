# **危険性その② デッドロック**

---

## **デッドロックとは**

<span style ="color: red;">**デッドロック**</span> とは、  
複数のスレッドが互いに相手のリソース解放を待ち続け、  
どちらも先に進めなくなる状態を指す。  
プログラムがフリーズしたように見えるが、エラーは発生しない為  
原因に気づきにくい。

さきほど紹介したミューテックスを複数組み合わせた際に特に発生しやすい。

---

## **デッドロックが起きるコード例**

二つのスレッドが二つのミューテックスを逆順にロックしようとする例。

```c++
#include <iostream>
#include <thread>
#include <mutex>

std::mutex g_mutexA;
std::mutex g_mutexB;

void ThreadA()
{
    g_mutexA.lock();                                     // 1. A をロック
    std::cout << "スレッドA: mutexA をロック" << std::endl;

    g_mutexB.lock();                                     // 3. B を待機し続ける
    std::cout << "スレッドA: mutexB をロック" << std::endl;

    g_mutexB.unlock();
    g_mutexA.unlock();
}

void ThreadB()
{
    g_mutexB.lock();                                     // 2. B をロック
    std::cout << "スレッドB: mutexB をロック" << std::endl;

    g_mutexA.lock();                                     // 4. A を待機し続ける
    std::cout << "スレッドB: mutexA をロック" << std::endl;

    g_mutexA.unlock();
    g_mutexB.unlock();
}

int main()
{
    std::thread tA(ThreadA);
    std::thread tB(ThreadB);

    tA.join(); // 永遠に終わらない
    tB.join();

    return 0;
}
```

処理の流れを追うと次のようになる。

```
スレッドA: mutexA をロックする
スレッドB: mutexB をロックする
スレッドA: mutexB をロックしようとする → スレッドBが持っている為、待機
スレッドB: mutexA をロックしようとする → スレッドAが持っている為、待機
```

スレッドAはBが `mutexB` を解放するのを待ち、  
スレッドBはAが `mutexA` を解放するのを待っている。  
しかし両者とも待機中なので解放は永遠に起きない。  
これがデッドロックとなる。

---

## **回避方法**

### **ロック対象や順番を正しく把握しておく事**

具体的な対策例として、ロック順番を順守するするなどが挙げられはする。     
上記の例では、スレッドBも `mutexA → mutexB` の順でロックすれば  
デッドロックは発生しない。

ただ、そもそも<span style ="color: red;">ロックを複数重ねないといけない状況になる事自体を避けた方が良い</span>。  
ロックしなくても安全な方法はないのか？を先に考える事をお勧めする。

デッドロックはミューテックスの使い方を誤ることで発生する。  
ロック順を上手くコントロールするか  
そもそも順序管理が不要な設計を心がける事が重要となる。
