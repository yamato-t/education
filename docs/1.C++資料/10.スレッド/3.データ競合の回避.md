# **データ競合の回避**

---

## **クリティカルセクションを守る**

前章で説明した通り、データ競合はクリティカルセクションに  
複数のスレッドが同時にアクセスすることで発生する。  
これを防ぐには、ある時点でクリティカルセクションに  
<span style ="color: red;">**一つのスレッドしかアクセスできない様にする**</span> 事が必要となる。

クリティカルセクションの安全性を確保する手段は多岐にわたり、  
スピンロック、セマフォ、読み書きロック、ロックフリーアルゴリズムなど  
様々な方法が存在する。  
今回はその入口として、C++ 標準ライブラリが提供する  
`std::mutex` と `std::atomic` を取り上げる。

---

## **ミューテックス**

<span style ="color: red;">**ミューテックス（Mutex）**</span> とは、  
Mutual Exclusion（相互排他）の略で、  
同時に一つのスレッドしかロックを取得できない仕組みを指す。

`<mutex>` ヘッダに含まれる `std::mutex` を使う。

```c++
#include <iostream>
#include <thread>
#include <mutex>

int g_counter = 0;
std::mutex g_mutex; // ミューテックスを用意する

void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        g_mutex.lock();   // ロックを取得（他スレッドは待機）
        ++g_counter;      // クリティカルセクション
        g_mutex.unlock(); // ロックを解放
    }
}

int main()
{
    std::thread tA(Increment);
    std::thread tB(Increment);

    tA.join();
    tB.join();

    // 正しく 200000 になる
    std::cout << "カウンタ: " << g_counter << std::endl;

    return 0;
}
```

`lock()` を呼んだスレッドだけがクリティカルセクションに入れる。  
別のスレッドが `lock()` を呼ぶと、先のスレッドが `unlock()` するまで待機する。

---

## **lock_guard**

`lock()` と `unlock()` を手動で呼ぶと、  
例外が発生した場合や途中で `return` した場合に  
`unlock()` を呼び忘れるリスクがある。  
`unlock()` が呼ばれないと他のスレッドが永遠に待ち続けるデッドロックが起きる。

この問題を防ぐ為に <span style ="color: red;">**std::lock_guard**</span> を使う。  
スコープを抜ける際に自動でロックを解放してくれる。

```c++
#include <iostream>
#include <thread>
#include <mutex>

int g_counter = 0;
std::mutex g_mutex;

void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        // スコープを抜けると自動的に unlock される
        std::lock_guard<std::mutex> lock(g_mutex);
        ++g_counter;
    }
}

int main()
{
    std::thread tA(Increment);
    std::thread tB(Increment);

    tA.join();
    tB.join();

    std::cout << "カウンタ: " << g_counter << std::endl;

    return 0;
}
```

`std::lock_guard` は RAII（Resource Acquisition Is Initialization）  
という手法を使っており、構築時にロックを取得し、  
破棄時に自動でロックを解放する。  
手動で `unlock()` を呼ぶ必要がなくなる為、安全性が高い。

---

## **アトミック操作**

ミューテックスは強力だが、ロックの取得・解放にコストがかかる。  
単純な数値の読み書きのみが目的であれば、  
<span style ="color: red;">**std::atomic**</span> を使う方が効率的となる。

```c++
#include <iostream>
#include <thread>
#include <atomic>

// int の代わりに atomic<int> を使う
std::atomic<int> g_counter(0);

void Increment()
{
    for (int i = 0; i < 100000; ++i)
    {
        ++g_counter; // アトミック操作で安全にインクリメント
    }
}

int main()
{
    std::thread tA(Increment);
    std::thread tB(Increment);

    tA.join();
    tB.join();

    std::cout << "カウンタ: " << g_counter << std::endl;

    return 0;
}
```

`std::atomic<T>` は、指定した型 `T` に対する読み書きを  
分割不可能な一つの操作（アトミック操作）として実行する事を保証する。  
ミューテックスよりも低コストで動作する為、  
単純なカウンタやフラグに向いている。

---

## **ミューテックスとアトミックの使い分け**

| 項目 | std::mutex | std::atomic |
|---|---|---|
| 向いている用途 | 複数の操作をまとめて保護したい場合 | 単一の変数の読み書きのみ |
| コスト | 高め | 低め |
| 対象 | どんな処理でも | 整数・ポインタなど限られた型 |

複数の変数をセットで更新する処理や、  
条件分岐を含む複雑なクリティカルセクションには  
`std::mutex` と `std::lock_guard` を使う。  
シンプルなカウンタやフラグの更新には `std::atomic` が適している。
