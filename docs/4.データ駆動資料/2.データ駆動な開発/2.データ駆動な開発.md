# **データ駆動**

---

## **企画者にとってのプログラミングとは**

企画者の方々は「プログラミングは難しそう」「エンジニアの仕事でしょ？」と思われるかもしれない。  
しかし、データ駆動の考え方を理解することで、企画とプログラムの橋渡しができ、  
より効率的で柔軟なゲーム開発が可能になる。

データ駆動がなぜ重要なのか、そしてどのように活用できるのかを、  
実際に手を動かしながら理解してもらう。

---

## **データ駆動とは何か？**

### **データ駆動の基本概念**

データ駆動（Data-Driven）とは、  
<span style="color: red;">**プログラムのロジック（処理）とデータ（設定値）を分離し、**</span>  
<span style="color: red;">**データを変更することでゲームの動作を制御する**</span>  
プログラミング手法の事。


**従来の方法**
```
敵の体力 = 100（プログラムに直接書かれている）
敵の攻撃力 = 20（プログラムに直接書かれている）
敵の移動速度 = 5（プログラムに直接書かれている）
```

このように、プログラム内に直接数値が記述されている物を**ハードコーディング**と呼ぶ。  
ハードコーディングの場合、エンジニアにしか数値調整が出来ない。

**データ駆動の基本をおさえた方法**
```
敵の設定ファイル（CSV、JSON等）:
名前, 体力, 攻撃力, 移動速度
スライム, 100, 20, 5
ゴブリン, 150, 35, 8
ドラゴン, 1000, 100, 3
```

つまり、プログラム内に直接記述せず、  
**外部ファイルに数値を記述してプログラムで利用する**

### **身近な例**


| 方式 | 特徴 | 変更時の対応 |
|------|------|-------------|
| ハードコーディング方式 | シェフが料理名と価格を覚えていて、お客さんに口頭で伝える | 価格変更時はシェフ全員に新しい情報を教える必要がある |
| データ駆動方式 | メニュー表（データ）を作成し、シェフはそれを見て対応 | 価格変更時はメニュー表だけを更新 |

ゲーム開発でも同じである。  
設定をデータ化することで、プログラムを変更せずに様々な調整が可能になる。

## **なぜデータ駆動が重要なのか？**

### **開発効率の向上**

**問題：バランス調整のたびにプログラム修正**

- 「このボスが強すぎる、体力を800に下げよう」
- プログラマーに依頼 → コード修正 → ビルド → テスト
- 1回の調整に数十分〜数時間

**解決：データ駆動による即座の調整**

- CSVファイルの数値を変更 → 即座に反映
- 企画者自身が調整可能
- 1回の調整に数秒〜数分

### **チーム協業の改善**

**従来の問題点**

- 企画者：「敵をもう少し弱くして」
- プログラマー：「どの値をどのくらい？」
- 企画者：「実際に見てみないと決められない...」

**データ駆動の利点**

- 企画者が直接データを編集
- リアルタイムで結果を確認
- プログラマーは機能実装に集中

### **品質向上とリスク軽減**

<span style="color: red;">**コード修正のリスク**</span>

- バランス調整のためのコード変更でバグが混入
- 関係ない部分に影響を与える可能性
- デグレード（既存機能の劣化）のリスク

<span style="color: red;">**データ変更の安全性**</span>

- データ変更は既存のロジックに影響しない
- バグが入りにくい
- 元の設定に簡単に戻せる

### **運営・アップデートの柔軟性**

**リリース後の調整**

- イベント時の特別設定
- ユーザーフィードバックに基づく調整
- A/Bテストによる最適化

これらがデータ駆動なら、アプリの更新なしに実現可能。

## **Unityでデータ駆動を実現する機能**

### **ScriptableObject**

**概要**  
UnityのScriptableObjectは、データを保存するためのクラス。  
Inspector上で編集でき、アセットとして管理できる。

**使用例：敵キャラクターの設定**
```csharp
[CreateAssetMenu(fileName = "EnemyData", menuName = "Game/EnemyData")]
public class EnemyData : ScriptableObject
{
    public string enemyName;
    public int health;
    public int attackPower;
    public float moveSpeed;
    public Sprite enemySprite;
}
```

**プレハブのInspectorパラメータとの違い**

よく質問される点として「プレハブのInspectorで直接パラメータを設定するのと何が違うのか？」がある。  
重要な違いを説明する：

<span style="color: red;">**プレハブ方式の問題点**</span>

- **データの重複**：同じ設定の敵が100体いると、同じデータが100回保存される
- **一括変更の困難**：全てのスライムの体力を変更するには、全プレハブを個別に修正
- **バリエーション管理**：「強いスライム」「弱いスライム」でプレハブが増殖
- **メモリ効率**：不要なデータがメモリに残る

<span style="color: red;">**ScriptableObject方式の利点**</span>

- **データの一元管理**：1つのScriptableObjectを複数のオブジェクトで共有
- **効率的な変更**：データを1箇所変更すれば全ての参照先に反映
- **柔軟なバリエーション**：同じプレハブに異なるデータを割り当て
- **メモリ効率**：必要なデータのみロード

**具体例**
```
従来：スライムプレハブ × 10体 = データ10個分のメモリ使用
ScriptableObject：スライムプレハブ × 10体 = データ1個分のメモリ使用（共有）
```

### **JSON/CSVファイル読み込み**

**概要**  
外部ファイルからデータを読み込む方法である。  
ExcelやGoogleスプレッドシートで編集できる。

**JSONの例**
```json
{
  "enemies": [
    {"name": "スライム", "health": 100, "attack": 20, "speed": 5},
    {"name": "ゴブリン", "health": 150, "attack": 35, "speed": 8}
  ]
}
```

**利点**

- 企画者が慣れ親しんだツール（Excel等）で編集可能
- 大量のデータを効率的に管理
- 他のツールとの連携が容易

### **Visual Scripting（旧Bolt）**

**概要**  
UnityのVisual Scriptingは、コードを書かずにノードベースでロジックを作成できるツール。  
データ駆動の観点から非常に有用である。

**データ駆動での活用**

- **設定可能なAI行動**：敵の行動パターンをデータとして保存
- **動的なイベント処理**：クエストやイベントの条件をビジュアルで設定
- **企画者による直接編集**：プログラミング知識なしでロジック調整

**利点**

- 企画者が直接ゲームロジックを編集可能
- 複雑な条件分岐もビジュアルで理解しやすい
- データとロジックの境界が明確

### **Timeline**

**概要**  
UnityのTimelineは、時間ベースのイベントやアニメーションを管理するツールである。

**データ駆動での活用**

- **カットシーンのデータ化**：演出をTimelineアセットとして保存
- **イベントシーケンス**：ボス戦の段階的な行動パターン
- **動的な演出切り替え**：プレイヤーの行動に応じた演出変更

**利点**

- 演出の調整が直感的
- プログラマーの介入なしで複雑なシーケンス作成
- データとして保存・共有可能

## **ハンズオン①：敵キャラクター管理システム**

実際に手を動かして、データ駆動の有用性を体験。

### **従来方式の問題を体験**

まずハードコーディングされた敵キャラクターのスクリプトを確認する。

```csharp
public class EnemyHardCoded : MonoBehaviour
{
    private void Start()
    {
        // スライムの設定（ハードコーディング）
        if (gameObject.name == "Slime")
        {
            GetComponent<Enemy>().health = 100;
            GetComponent<Enemy>().attackPower = 20;
            GetComponent<Enemy>().moveSpeed = 5f;
        }
        // ゴブリンの設定（ハードコーディング）
        else if (gameObject.name == "Goblin")
        {
            GetComponent<Enemy>().health = 150;
            GetComponent<Enemy>().attackPower = 35;
            GetComponent<Enemy>().moveSpeed = 8f;
        }
    }
}
```

<span style="color: red;">**問題点**</span>

1. 新しい敵を追加するたびにコード修正が必要
2. バランス調整のたびにプログラマーに依頼が必要
3. 設定値がコード内に散らばって管理が困難

### **データ駆動方式の実装**

次に、ScriptableObjectを使ったデータ駆動方式を実装。

**ステップ①: データクラスの作成**
```csharp
[CreateAssetMenu(fileName = "EnemyData", menuName = "Game/EnemyData")]
public class EnemyData : ScriptableObject
{
    [Header("基本情報")]
    public string enemyName;
    public Sprite enemyIcon;
    
    [Header("戦闘パラメータ")]
    public int health = 100;
    public int attackPower = 20;
    public float moveSpeed = 5f;
    
    [Header("報酬")]
    public int expReward = 50;
    public int goldReward = 10;
}
```

**ステップ②: データ駆動な敵クラス**
```csharp
public class EnemyDataDriven : MonoBehaviour
{
    [SerializeField] private EnemyData enemyData;
    private int currentHealth;
    
    private void Start()
    {
        ApplyEnemyData();
    }
    
    private void ApplyEnemyData()
    {
        if (enemyData == null) return;
        
        currentHealth = enemyData.health;
        GetComponent<Enemy>().attackPower = enemyData.attackPower;
        GetComponent<Enemy>().moveSpeed = enemyData.moveSpeed;
        
        // UIの更新
        transform.Find("NameText").GetComponent<Text>().text = enemyData.enemyName;
        transform.Find("Icon").GetComponent<Image>().sprite = enemyData.enemyIcon;
    }
}
```

### **実際の利用例**

1. **新しい敵の追加**  

   右クリック → Create → Game → EnemyData  
   新しいEnemyDataアセットを作成  
   Inspector上で各パラメータを設定  
   既存の敵オブジェクトにアタッチ  
   
   → **コード修正なしで新しい敵が完成する**

2. **バランス調整**  

   EnemyDataアセットを選択  
   Inspectorで体力や攻撃力を調整  
   Playモードで即座に反映確認  
   
   → **プログラマーに依頼せずに調整が完了する**

3. **一括設定**  

   複数の敵オブジェクトに同じEnemyDataを設定  
   EnemyDataを修正  
   全ての敵に変更が反映される  
   
   → **効率的な管理が可能**

### **ハンズオン②：クエストシステムの実装**

**ステップ①: クエストデータクラス**
```csharp
[CreateAssetMenu(fileName = "QuestData", menuName = "Game/QuestData")]
public class QuestData : ScriptableObject
{
    [Header("基本情報")]
    public string questTitle;
    [TextArea(3, 5)]
    public string description;
    public Sprite questIcon;
    
    [Header("条件")]
    public QuestType questType;
    public string targetName;  // 倒すべき敵の名前 or 集めるアイテム名
    public int requiredAmount;
    
    [Header("報酬")]
    public int expReward;
    public int goldReward;
    public string rewardItem;
}

public enum QuestType
{
    KillEnemy,    // 敵を倒す
    CollectItem,  // アイテム収集
    ReachLocation // 場所到達
}
```

**ステップ②: データ駆動なクエストマネージャー**
```csharp
public class QuestManager : MonoBehaviour
{
    [SerializeField] private QuestData[] availableQuests;
    [SerializeField] private Transform questListParent;
    [SerializeField] private GameObject questUIPrefab;
    
    private void Start()
    {
        GenerateQuestUI();
    }
    
    private void GenerateQuestUI()
    {
        foreach (QuestData quest in availableQuests)
        {
            GameObject questUI = Instantiate(questUIPrefab, questListParent);
            questUI.transform.Find("Title").GetComponent<Text>().text = quest.questTitle;
            questUI.transform.Find("Description").GetComponent<Text>().text = quest.description;
            questUI.transform.Find("Requirement").GetComponent<Text>().text = 
                $"{quest.targetName} × {quest.requiredAmount}";
            questUI.transform.Find("Reward").GetComponent<Text>().text = 
                $"EXP: {quest.expReward}, Gold: {quest.goldReward}";
        }
    }
}
```

### **実際の利用例**

1. **新クエスト追加**：QuestDataアセット作成でクエスト追加
2. **条件変更**：必要数や対象の変更が即座に反映
3. **報酬バランス調整**：プログラム修正なしで報酬変更
4. **季節イベント対応**：期間限定クエストの簡単な追加・削除

**Visual Scriptingとの連携**  

後の講義で紹介する **Visual Scripting** のグラフ自体をScriptableObjectとして保存し、  
敵ごとに異なる AI 行動を割り当てるといった応用も可能。  
これにより、プログラムを書かずに敵の行動パターンをデータとして管理できる。

## **まとめ：企画者が知っておくべきポイント**

### **データ駆動導入のメリット再確認**

1. **開発速度の向上**：調整作業の大幅な時短
2. **品質向上**：バグリスクの軽減
3. **チーム効率**：企画とプログラムの分業最適化
4. **運営柔軟性**：リリース後の迅速な対応

### **企画者として心がけること**

<span style="color: red;">**データ設計への参画**</span>

- どんな項目が必要か積極的に提案
- 将来の拡張性を考慮した設計
- プログラマーとの密な連携

<span style="color: red;">**データ管理の責任**</span>

- データの整合性を保つ
- 変更履歴の管理
- チームメンバーへの情報共有

<span style="color: red;">**ツールの活用**</span>

- Visual Scriptingでロジックの可視化
- Timelineで演出のデータ化
- 各ツールの特性を理解した使い分け

<span style="color: red;">**継続的な改善**</span>

- データ駆動の効果測定
- より良いワークフローの模索
- 新しいツールの積極的な導入

### **余談**

**データ駆動にすると開発コストが増えないか？**  
初期投資は必要だが、中長期的には大幅なコスト削減になる。  
実際の現場ではまず間違いなくこういった仕組みが利用されている。  
特に運営フェーズでの効果は絶大。

**全てをデータ駆動にする必要があるのか？**  
そんなことはない。  
無理やり使うのは正しいとは言えない。  
変更頻度の高い部分から段階的に導入するのが効率的。

## **おわりに**

データ駆動は単なるプログラミング手法ではない。  
ゲーム開発全体の効率化と品質向上を実現する、現代のゲーム開発に不可欠な考え方。

企画者の皆さんがこの考え方を理解し実践することで、より良いゲーム作りが可能になる。  
最初は小さなところから始めて、徐々に適用範囲を広げていってほしい。
