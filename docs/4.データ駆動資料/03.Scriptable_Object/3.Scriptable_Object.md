
# **Scriptable Object**  

---  

## **Scriptable Object とは**  

Scriptable Object は Unity におけるアセット型のデータコンテナであり、  
シーンや GameObject に依存しない独立データとして存在する。  

ゲーム動作時の参照は軽量で、メモリの消費量を抑え、  
同一データを複数オブジェクトで共有できる。  

<span style ="color: red;">**シーン非依存、共有、再利用、差し替えの容易さが核心**</span>  

### **Scriptable Object を使わない場合（ハードコーディング）**  

もし Scriptable Object を使わず、MonoBehaviour 内で直接パラメータを  
定義すると以下のようになる。  

```csharp
using UnityEngine;
using UnityEngine.UI;

public class HardcodedItemDisplay : MonoBehaviour
{
    [SerializeField] private Text nameText;
    [SerializeField] private Image iconImage;
    [SerializeField] private Text priceText;

    void Start()
    {
        string itemName = "Potion";
        int price = 50;
        // アイコンは Project 内のリソースを直参照するなど

        nameText.text = itemName;
        priceText.text = price.ToString() + " G";
    }
}
```

この方法では値をコードに埋め込むため、  
変更するたびにコード修正と再ビルドが必要となる。  
また、複数オブジェクトで同じ値を共有するのが難しく、  
アイテム数が増えると管理コストが爆発的に増加する。  

<span style ="color: red;">**Scriptable Object はこの欠点を解決する手段である**</span>  

主な用途は下記の通り。

- キャラクターステータスやアイテム定義などの静的データ  
- 難易度やドロップ率などのゲーム全体設定  
- サウンドやエフェクトの目次  
- 処理の雛形（戦略パターンやコマンドパターンの実体）

## **作成と基本利用**  

### **クラス定義とアセット化**  

下記はアイテムデータの例。

```csharp
using UnityEngine;

[CreateAssetMenu(fileName = "NewItemData", menuName = "Game/ItemData")]
public class ItemData : Scriptable Object
{
    public string itemName;
    public Sprite icon;
    public int price;
}
```

CreateAssetMenu 属性により Project ウィンドウから生成可能になる。  
アセットは .asset として保存され、シーン横断で参照可能となる。  

### **参照と表示**

作成したアイテムデータアセットを参照する例。

```csharp
using UnityEngine;
using UnityEngine.UI;

public class ItemObject : MonoBehaviour
{
    public ItemData itemData;
    [SerializeField] Text nameText;
    [SerializeField] Image iconImage;
    [SerializeField] Text priceText;

    void Start()
    {
        if (itemData == null) return;
        nameText.text = itemData.itemName;
        iconImage.sprite = itemData.icon;
        priceText.text = itemData.price.ToString() + " G";
    }
}
```

アイテムオブジェクト側は参照を保持するのみで、生成コストは増えない。  
Prefab、シーン、複数コンポーネントで同じデータを一貫して利用できる。

## **共有を保ちつつ個別に変える**  

同一のデータアセットを参照している全オブジェクトが同じ値を見る事になる。  
ただ、内容によって「一部だけパラメータを変化させたい」といった状況もあるはず。  

その場合、基本的に下記の二つの方法で検討する。  

### **Instantiate 方式（ランタイムコピー）**  

データアセットを別で生成（コピー）してしまう方法。

```csharp
public class Enemy : MonoBehaviour
{
    [SerializeField] CharacterData baseData;  // 共有
    CharacterData runtimeData;                // 個別

    void Awake()
    {
        runtimeData = Instantiate(baseData);
    }

    void Start()
    {
        runtimeData.hp += 50; // この個体だけ強化
    }
}
```

コピーはこのオブジェクト専用の実体となり、  
元のアセットには一切影響しない。  
バリエーション違いのオブジェクトなどで有用。  

### **差分（Override）方式**  

```csharp
[System.Serializable]
public class CharacterOverride
{
    public bool overrideHp;
    public int hp;
    public bool overrideAttack;
    public int attack;
}

public class Enemy : MonoBehaviour
{
    [SerializeField] CharacterData baseData;
    [SerializeField] CharacterOverride overrideData;

    public int HP => overrideData.overrideHp ? overrideData.hp : baseData.hp;
    public int Attack => overrideData.overrideAttack ? overrideData.attack : baseData.attack;
}
```

共有データを基準値として扱い、必要部分のみを局所的に上書きする。  
バフ、デバフといった、特定の条件で特定のパラメータだけ変更する等で有用。

### **ハンズオン**  

1. CharacterData を作成し、HP=100、攻撃=10 を基準値とする。  
2. 敵を二体生成し、一体は Instantiate 方式で HP+50、他方は差分で攻撃=20 を適用。  
3. 10 体生成し、半数をコピー方式、半数を差分方式で混在運用し、ログで検証する。

## **プログラミング応用**  

Scriptable Object は単なるデータだけではなく、  
振る舞いや動作も含める事が出来る。

プログラミングにおける「継承」を利用する。

### **抽象的な動作**  

```csharp
// スキルを抽象的（曖昧）に定義
public abstract class SkillBase : Scriptable Object
{
    public string skillName;
    public Sprite icon;
    public abstract void Activate(GameObject user);
}

// スキル「ファイアーボール」（具体的）を定義
[CreateAssetMenu(fileName = "NewFireball", menuName = "Game/Skills/Fireball")]
public class FireballSkill : SkillBase
{
    public int damage;
    public override void Activate(GameObject user)
    {
        Debug.Log($"{skillName} ダメージ {damage}");
    }
}

// スキル「パワーアップ」（具体的）を定義
[CreateAssetMenu(fileName = "NewPowerUp", menuName = "Game/Skills/PowerUp")]
public class FireballSkill : SkillBase
{
    public int powerUp;
    public override void Activate(GameObject user)
    {
        Debug.Log($"{skillName} バフ {powerUp}");
    }
}

```

抽象的に「動作」を定義し、派生先で具体的に内容を定義する。  
配列を SkillBase[] にする事で、曖昧な定義の変数配列に、具体的な定義を設定していく事が出来る。  
所謂「多態的な設定」が行える。

### **インターフェース**  

抽象的と似た方法として、インターフェイスというものがある。  
抽象的な場合はメンバ変数を持ち、「動作」だけではなく「パラメータ」も保持していたが、  
インターフェイスの場合は「動作」のみを持ち、データは全て派生先に持たせる。  
その為基底クラスが Scriptable Object を継承しない。

これにより、「動作（振る舞い）」と「パラメータ（データ）」を分離し、  
それぞれの役割を明確にする事が出来る。


```csharp
// 「使う」という動作を表すインターフェイス
public interface IUsable { void Use(GameObject user); }


// 「使う」という動作を行う事が出来るアイテムクラス
[CreateAssetMenu(menuName = "Game/Items/Potion")]
public class PotionItem : Scriptable Object, IUsable
{
    public int heal;
    public void Use(GameObject user) => Debug.Log($"HP +{heal}");
}
```

横断的な機能要件を満たせる。  
「使えるもの」集合の抽象化に有効。  

### **SerializeReference で柔軟な多態シリアライズ**  

通常の Unity シリアライズは抽象やインターフェースのリストを扱えない。  
SerializeReference を用い、実体は派生型、参照は基底型で保持する。  

```csharp
// 「敵の行動」に「実行する」という抽象的（曖昧）な定義を用意する
[System.Serializable]
public abstract class EnemyAction { public abstract void Execute(GameObject enemy); }

// 曖昧に定義していた「実行する」という行動を「巡回する」という具体的な定義に上書きする
[System.Serializable]
public class PatrolAction : EnemyAction
{
    public float speed;
    public override void Execute(GameObject enemy) { Debug.Log($"巡回 {speed}"); }
}

// 曖昧に定義していた「実行する」という行動を「攻撃する」という具体的な定義に上書きする
[System.Serializable]
public class AttackAction : EnemyAction
{
    public int power;
    public override void Execute(GameObject enemy) { Debug.Log($"攻撃 {power}"); }
}

// 「敵の行動」を抽象的（曖昧）な形で保持する
[CreateAssetMenu(menuName = "Game/EnemyAI")]
public class EnemyAI : Scriptable Object
{
    [SerializeReference] public System.Collections.Generic.List<EnemyAction> actions;
}
```

Inspector 上で複数の「具体的な行動」を同一リストで保持する事が出来るようになる。  
順序変更や差し替えも容易。

## **処理そのものを Scriptable Object に収める**  

データだけでなく処理も Scriptable Object に置ける。  

### **アクションシステム**  

```csharp
// Scriptable Object で「ゲーム上のアクション」を保持し、アクションの内容は曖昧にする
public abstract class GameAction : Scriptable Object
{
    public abstract void Execute(GameObject actor);
}

// 「ゲーム上のアクション」を「ジャンプ」という具体的な行動に上書きする
[CreateAssetMenu(menuName = "Game/Actions/Jump")]
public class JumpAction : GameAction
{
    public float jumpForce = 5f;
    public override void Execute(GameObject actor)
    {
        var rb = actor.GetComponent<Rigidbody>();
        rb?.AddForce(Vector3.up * jumpForce, ForceMode.Impulse);
    }
}
```

### **利用側の組み立て**  

```csharp
// プレイヤーに「ゲーム上のアクション」を持たせ、スペースキーで実行する
public class Player : MonoBehaviour
{
    [SerializeField] GameAction[] actions;
    void Update()
    {
        if (Input.GetKeyDown(KeyCode.Space))
            // actions[0] が「ジャンプ」なら、ジャンプの挙動を実行する
            actions[0].Execute(gameObject);
    }
}
```

新しい挙動を追加したい場合はアセットを作って配列に差し込むだけ。  
ビルド不要の調整速度が確保できる。

## **「行動」を Scriptable Object で持たせる場合の原則**  

アセットは共有であるため、内部で何らかの状態を持ち、かつ更新されてしまうと動作が変わる恐れがある。  
その為次の原則を置く。  

- Scriptable Object は「行動の定義のみ」を徹底し、「状態」を表すメンバは持たない
- 「行動」に何らかのパラメータが必要な場合は、メンバ変数ではなく引数で渡す
- 個別のパラメータをどうしてもメンバ変数で持ちたい場合、コピーして新しい物を作る


### **外部状態オブジェクトの注入**  

```csharp
// 状態を分離
public class CharacterState
{
    public int combo;
    public float stamina;
}

// AttackAction は「状態」を表すメンバ変数を持たず、「行動」を表す関数だけ保持する
[CreateAssetMenu(menuName = "Game/Actions/Attack")]
public class AttackAction : Scriptable Object
{
    public int damage;
    public void Execute(GameObject actor, CharacterState state)
    {
        state.combo++;
        Debug.Log($"攻撃 {damage} / コンボ {state.combo}");
    }
}
```

### **複製で個体差を内包**  

```csharp
var action = Scriptable Object.Instantiate(originalAction);
```

動作中専用に個別の実体を作成し、  
安全に可変データを保持できる。

## **どういったパターンに有用か**  

- **AI 行動リスト**  

  SerializeReference でアクション列を持ち、  
  Timer 条件や距離条件などのデコレータを組み合わせる。  

- **スキルツリー**  

  抽象 SkillBase と派生スキルをアセット化し、  
  前提関係やコストをデータで表現する。  

- **アイテム拡張**  

  IUsable、IEquippable などのインターフェースで多面化し、  
  UI 生成はアセットメタ情報から自動化する。  

- **データ駆動 UI**  

  設定アセットを元に動的にメニューを生成し、  
  ビルドレスで項目追加と文言変更に追随する。  

---  


## **まとめ**  

Scriptable Object はデータと処理の両方をアセット化し、  
シーンに依存しない共有と差し替えを可能にする。  
プログラム基準ではなく、データ基準で動作や設定を任意に組み替える事が出来るようになる。

「企画者のみで動作や設定を変更出来る」という事は、  
「ゲーム作りのトライアンドエラー、クラッシュアンドビルドがやりやすい」という事でもあり  
「より面白い物を検証しやすい」という事でもある。

<span style ="color:  
red;">**変更に強いゲーム開発を実現出来る様に意識してほしい**</span>
