# **企画者とエンジニアの協業**
## **チームにおける意思疎通の重要性**

ゲーム開発はチーム開発であり、  
考え方も得意分野も違う色々な人員が集まって一つの製品を作り上げていく。

特に企画は各役職同士の橋の役割をする機会が多い。  
つまり企画は**どのセクションとも上手く連携出来なければならない**。

前期はゲームプログラムの講義として、  
プログラムそのものではなく**要件定義**として

- 必要な要素を如何に分解して整理するか
- エンジニアへの上手な依頼の仕方
- エンジニアとの連携をとる為にエンジニアリングを知る

をテーマに掲げてやってきた。  

---

## **要件定義の重要性**

### **要件定義とは何か**

要件定義とは、「何を作るのか」を明確に定義する工程。  
これは建築でいう設計図のようなもので、  
ここが曖昧だと後々大きな問題となる。  

ゲーム開発における要件定義では、  
以下の要素を明確にする必要がある。  

<span style ="color: red;">**機能要件**</span>  
- プレイヤーができること（アクション、移動、アイテム使用など）  
- システムの動作（セーブ・ロード、スコア計算、レベルアップなど）  
- UI/UXの具体的な振る舞い  

<span style ="color: red;">**非機能要件**</span>  
- 対応プラットフォーム（PC、モバイル、コンソールなど）  
- 想定プレイ人数と同時接続数  
- レスポンス時間の目標値  
- メモリ使用量の制限  

### **要件定義の良し悪しの例**

「ジャンプアクション」を実装する場合の比較

| 項目 | 悪い例 | 良い例 |
|------|--------|--------|
| **基本要求** | プレイヤーがジャンプ出来る様に実装 | 以下の詳細仕様でジャンプ機能を実装 |
| **入力** | 未定義 | スペースキーまたはAボタンの押下 |
| **実行条件** | 未定義 | 地面に接触している時のみ実行可能 |
| **物理パラメータ** | 未定義 | 初速度15m/s、重力加速度9.8m/s² |
| **着地判定** | 未定義 | プレイヤーの足元コライダーが地面タグと接触 |
| **アニメーション** | 未定義 | ジャンプ開始、空中、着地の3段階 |
| **音響効果** | 未定義 | ジャンプ音、着地音をそれぞれ再生 |
| **制約事項** | 未定義 | 連続ジャンプ不可、空中での慣性考慮 |

良い要件定義では、  
エンジニアが実装に必要な情報がすべて明確に定義されている。  

### **要件定義における企画者の責任**

企画者は「何を作りたいか」を伝えるだけでは不十分。  

- 「なぜそれが必要なのか」  
- 「プレイヤーにどんな体験を提供したいのか」  

という背景も共有する必要がある。  
これにより、エンジニアも単なる作業者ではなく、  
ゲームを一緒に作り上げるチームの一員として機能する。

## **エンジニアとの認識合わせの重要性**

### **技術的制約の理解**

ゲーム開発では、企画職であっても技術的制約を理解する必要が出てくる。  
これを知らない企画者がよく陥る問題として、  
「技術的には不可能」または「非常にコストの高い」  
要求を出してしまう事が挙げられる。  

### **定期的なコミュニケーションの重要性**

<span style ="color: red;">**日々の報連相**</span>  
短時間でも良いので、進捗と課題を共有する事が重要。  
問題の早期発見につながる。  
問題の早期発見に繋がるという事は、早期解決の準備を整えられるという事でもある。

<span style ="color: red;">**プロトタイプでの確認**</span>  
仕様書だけでなく、  
実際に動くプロトタイプを使って認識を合わせることが重要。  
「百聞は一見に如かず」という言葉通り、  
実際の動作を見ることで多くの誤解が解消される。  
これも、比較的短い間隔で定期的に行うべき。

<span style ="color: red;">**用語の統一**</span>  
プロジェクト内で使用する用語を統一する。  
例えば「敵」「エネミー」「モンスター」を混在させると、  
混乱の原因となる。  

### **エンジニアの視点を理解する**

エンジニアは常に以下のことを考えている

- パフォーマンス（処理速度、メモリ使用量）  
- 保守性（後で修正しやすいコード）  
- 拡張性（機能追加に対応できる設計）  
- 再利用性（他の部分でも使えるコード）  

これらの観点をある程度理解した上で話す事で、  
より建設的な議論ができるようになる。

## **スケジュールと進捗管理の重要性**

### **推奨バッファ率**

ゲーム開発では予期せぬ技術的課題が頻繁に発生する。  
スケジュールにはバッファを設ける。  

| 作業種類 | 基本見積もり時間 | バッファ率（例） | 実際の計画時間 | 理由 |
|----------|------------------|----------------|----------------|------|
| **新規機能実装** | 10日 | 1.5倍 | 15日 | 未知の技術課題が多発 |
| **既存機能の改修** | 5日 | 1.3倍 | 6.5日 | 既存コードとの影響調査必要 |
| **バグ修正** | 2日 | 1.2倍 | 2.4日 | 根本原因の特定に時間を要する |
| **UI/UX実装** | 8日 | 1.4倍 | 11.2日 | デザイン調整の反復が必要 |
| **最適化・パフォーマンス調整** | 6日 | 1.6倍 | 9.6日 | 効果測定と再調整のサイクル |

各タスクのバッファを最後に集め、  
発生した余裕を前倒して進めていく「クリティカルチェーン」と呼ばれる方法もある。  
個人的にクリティカルチェーンはお勧め。  
余裕を含めた計画をたてると、余裕も含めて計画通りに進めてしまうため、  
各タスク作業後に無駄な時間が発生しやすい。

### **マイルストーンの設定**

大きな目標を小さな達成可能な単位に分割する。  

1. **プリプロダクション**：コア機能の検証  
2. **アルファ版**：主要機能の実装完了  
3. **ベータ版**：すべての機能実装、バランス調整開始  
4. **ゴールドマスター**：リリース準備完了  

各マイルストーンで明確な成果物と判定基準を設定することが重要。  
近年では**バーティカルスライス**と呼ばれる**製品版の完成度を持つ１ステージ**の実装を取り込む例も多い。  
製品版の完成度である為、見た目やゲームのパフォーマンスなどを含めて完成イメージを掴む事が出来る。  
ただし「製品版の完成度を開発前半に実現する」という意味でもある為、中々に過酷なスケジュールになりがち。


### **進捗可視化の工夫**

<span style ="color: red;">**ガントチャート**</span>  
作業予定を横並びに一覧化し、  
一目で「何を」「いつから始めて」「いつまでに終わらせる」が把握できる。  
予定通りに進んでいるかどうかの確認も行いやすい。

<span style ="color: red;">**バーンダウンチャート**</span>  
残作業量の推移を可視化し、  
現在のペースでデッドラインに間に合うかを判断できる。  

<span style ="color: red;">**カンバンボード**</span>  
タスクの状態（TODO、進行中、レビュー中、完了）を視覚的に管理できる。

## **必須の技術用語集**

企画者が日常的にエンジニアとやり取りする際に必要となる  
重要な技術用語を、カテゴリ別に整理した。  

### **パフォーマンス関連用語**

| 用語 | 意味 | 企画への影響 | 対策例 |
|------|------|-------------|--------|
| **フレームレート（FPS）** | 1秒間に描画される画面数 | 30FPS以下でカクつきを感じる | エフェクト数制限、オブジェクト数調整 |
| **処理落ち** | 処理能力不足によるFPS低下 | ゲームプレイに直接影響 | 負荷の高い演出の見直し |
| **スパイク** | 特定タイミングでの処理時間急増 | 一時的な動作停止 | 処理の分散実行、軽量化 |
| **メモリ負荷** | RAM使用量の増加 | クラッシュやスワップ発生 | アセットサイズ最適化 |
| **ガベージコレクション** | 不要メモリの自動回収処理 | 一時的な処理停止 | オブジェクト生成頻度の調整 |

### **描画・グラフィック関連用語**

| 用語 | 意味 | 企画への影響 | 最適化手法 |
|------|------|-------------|-----------|
| **ドローコール** | CPUからGPUへの描画命令 | 多いと処理速度低下 | バッチ、インスタンス |
| **テクスチャストリーミング** | 必要な画像の動的読み込み | ローディング時間増加 | 画像圧縮、LOD使用 |
| **LOD** | 距離に応じた詳細度変更 | 遠景の表現品質に影響 | 段階的モデル作成 |
| **オクルージョンカリング** | 見えない部分の描画省略 | 複雑なシーンでの最適化 | レベルデザインでの配慮 |

### **システム・アーキテクチャ関連**

<span style ="color: red;">**同期・非同期処理**</span>  
同期処理は前の処理完了を待ってから次を実行、  
非同期処理は待たずに並行実行。  
ローディングやネットワーク通信では非同期処理が重要である。  

<span style ="color: red;">**シーケンス**</span>  
一連の順序を持つ処理や演出を管理する仕組み。  
「プレイヤーがある挙動を開始して○フレーム目にxからエフェクトを出す」などの時間的な制御がしやすい。  
UE ではシーケンサー、Unity ではタイムラインという機能が該当する。

<span style ="color: red;">ステートマシン</span>
オブジェクトの状態と、その状態間の遷移を管理する仕組み。  
例としてプレイヤーの「待機 → 移動 → 攻撃 → 待機」といった動作の流れを制御できる。  
実装方法はコードによるものからノードベースのエディタまで様々で、エンジンによって異なる。  
代表的な形式は有限状態機械（FSM: Finite State Machine）。  
AI制御では、FSMとは別の手法としてビヘイビアツリー（Behavior Tree）も用いられる。  

<span style ="color: red;">**イベント駆動**</span>  
特定の出来事（イベント）をトリガーとして処理を実行する設計パターン。  
「HPが0になったら死亡処理を実行」のような制御に使用する。  

### **データ管理関連**

<span style ="color: red;">**シリアライゼーション**</span>  
オブジェクトの状態をファイルやネットワークで  
送信可能な形式に変換する処理。  
セーブデータの保存に使用される。  
逆は**デシリアライゼーション**。

## **認識齟齬による問題**

### **よくある認識齟齬パターン**

| 企画者の認識 | エンジニアの実情 | 発生する問題 | 
|-------------|------------------|-------------|
| 「数値変更は簡単」 | UI、バランス、DB設計への影響大 | 予想外の工数増加 |
| 「リアルタイム対応」 | 遅延許容度が不明確 | 技術選択の迷い |
| 「PCで動けばスマホでも」 | 処理能力、UI、操作性が全く異なる | 移植工数の大幅増加 |
| 「簡単な追加機能」 | 既存システムとの整合性確保が困難 | デグレードや不具合多発 |
| 「綺麗なエフェクト」 | パフォーマンスコストが高い | 処理落ちやメモリ不足 |

### **具体的な事例**

<span style ="color: red;">**①「簡単な修正」の齟齬**</span>  

**発端**  
企画者：「敵を強くしたいのでHPを 1000 から12000 にして」  

**問題**  
| 要素 | 企画者の想定 | 実際の影響 |
|------|-------------|-----------|
| **変更内容** | 敵のHPを1000→12000に変更 | 単純な数値変更 |
| **UI影響** | なし | 表示桁数変更でレイアウト崩れ |
| **バランス影響** | 敵が強くなるだけ | 他の敵との相対バランス崩壊 |
| **データ影響** | なし | セーブデータ互換性の確認必要 |
| **テスト影響** | なし | 全難易度でのバランステスト必要 |
| **実工数** | 5分 | 2-3日 |

<span style ="color: red;">**②「リアルタイム」の誤解**</span>  

**発端**  
企画者：「プレイヤー同士がリアルタイムで対戦できるようにして」  

**問題**  
「リアルタイム」という言葉の解釈が人によって異なる。  
ゲームジャンルによって許容される遅延も大きく違う。  

<span style ="color: red;">**③「プラットフォームの制約」**</span>  

**発端**  
企画者：「PCで動いているからスマートフォンでも同じように動かせるはず」  

**問題**  
プラットフォームごとの制約と特性を理解せずに  
移植を前提とした企画を立てることで、  
開発工数が大幅に増加。

## **協業の為に意識しておく事**

1. <span style ="color: red;">**説明は具体的に**</span>：曖昧な表現を避け、数値や条件を明確に示す  
2. <span style ="color: red;">**やり取りを継続する**</span>：定期的な情報共有を欠かさない  
3. <span style ="color: red;">**相互に理解する**</span>：エンジニアの視点と制約を理解する  
4. <span style ="color: red;">**透明性を持つ**</span>：問題や懸念を隠さずにオープンに議論する  
5. <span style ="color: red;">**柔軟な対処**</span>：技術的制約に応じた企画調整を受け入れる  

### **企画者が身に付けるべきスキル**

<span style ="color: red;">**技術リテラシー**</span>  
完全に理解する必要はないが、  
基本的な技術概念は把握しておくべき。  

<span style ="color: red;">**見積もり能力**</span>  
機能の複雑さと実装工数の関係を  
感覚的に理解できる様に。  

<span style ="color: red;">**優先順位付け**</span>  
限られたリソースの中で、  
最も価値の高い機能を見極める能力が重要。  

<span style ="color: red;">**リスク管理**</span>  
技術的な事、スケジュール的な事など、  
幅広い範囲の「リスク」を早期に発見、報告し、  
適切な対策を講じれる様に。

## **ディスカッション**

### **パフォーマンス問題を解決依頼**

**設定**  
アクションRPGで、  
画面上に50体の敵が同時出現するシーンがある。  
テストプレイでフレームレートが大幅に低下している。  

解決の為に、皆さんならどの様に行動するか。  
そもそもどの様に解決するか。

| 分析項目 | 可能性1 | 可能性2 | 可能性3 |
|---------|---------|---------|---------|
| **理由** | CPU処理過多 | GPU描画負荷 | メモリ不足 |
| **要因** | 敵AI計算50体分やってるから | 50体分のポリゴン描画してるから | テクスチャ読み込み多すぎるから |
| **症状** | 全体的な処理遅延 | 描画のみ重い | 一時的なフリーズ |
| **解決策** | AI処理の軽量化 | LOD導入、ドローコール削減 | アセット読込の軽量化や最適化 |
| **企画調整** | 敵の行動パターン簡素化 | 遠距離敵の簡略表示 | 敵種類の統一 |

**ディスカッション項目**  
1. どういった原因で負荷が挙がっていると思われるか？  
2. ゲーム体験を損なわない解決策は？  
3. 企画段階で防ぐことは出来るか？  

### **機能追加要求**

**設定**  
開発終盤で「プレイヤーが建物を自由に建築できる機能」の追加要求が出た。  
リリースまで2ヶ月しかないが、どう考えても半年はかかる。  
時間が足りない。  

解決の為に、皆さんならどの様に行動するか。  
そもそもどの様に解決するか。

| 評価軸 | フル実装 | 簡易実装 | 次期バージョン |
|--------|----------|----------|----------------|
| **開発期間** | 6ヶ月 | 3ヶ月 | リリース後6ヶ月 |
| **機能範囲** | 完全自由建築 | プリセット組み合わせ | フル機能 |
| **技術的課題** | 地形変更、物理演算 | UIとデータ管理のみ | 十分な検証時間あり |
| **品質リスク** | 高（デグレード多発） | 中（限定的影響） | 低（充分なテスト） |
| **プレイヤー満足度** | 高 | 中 | 高（完成度向上） |
| **推奨度** | × | △ | ○ |

**判断基準**  
- リリーススケジュールへの影響  
- 既存機能への悪影響リスク  
- プレイヤーにとっての価値  
- マーケティング戦略との整合性  


### **ユーザビリティと技術制約**

**設定**  
「ワンタップで既存 SNS サービスへの投稿」という、ユーザーにとってとても使いやすい機能を実装したいが、  
プラットフォームの機能制約により複雑な認証フローが必要で、ワンタップによる投稿が出来そうにない。  

解決の為に、皆さんならどの様に行動するか。  
そもそもどの様に解決するか。

**ディスカッション項目**  
1. ユーザビリティと実装コストのトレードオフ  
2. 段階的な機能提供の可能性  
3. プラットフォーム別の対応方針  

**検討要素**  
- ユーザーの利用頻度予測  
- 開発・保守コスト  
- セキュリティ考慮事項  
- 代替UI/UXパターン
