# **工数見積もり**
## **工数予測とスケジュール策定の基礎**

---

## **プランニングポーカーとは**

プランニングポーカーは、ゲーム開発プロジェクトにおいて、  
各タスクの工数を効率的かつ正確に見積もるためのチーム手法。  
アジャイル開発で広く使用されており、個人の偏見を排除し、  
チーム全体の知識と経験を活用して、より精度の高い見積もりを実現する。

### **プランニングポーカーの有効性**

ゲーム開発では、プログラミング、企画、アート制作、サウンド制作、  
レベルデザインなど、多岐にわたる専門分野が関わる。  
従来の見積もり方法では、声の大きい人や経験豊富な人の意見に引きずられがちだが、  
プランニングポーカーでは全員が同時に見積もりを提示するため、  
より民主的で多角的な視点を得ることができる。

例えば、「敵キャラクターのAI実装」というタスクがあった場合、  
プログラマーは技術的な複雑さを、企画者はゲームバランス調整の難しさや  
ユーザー体験への影響を、それぞれ異なる視点から評価できる。

### **ゲーム開発における具体的なメリット**

- <span style ="color: red;">**隠れた要件の発見**</span>
    - 異なる職種のメンバーが議論することで、当初見落としていた要件や依存関係が明らかになる
    - 例：「アイテムショップ機能」でプログラマーはデータベース設計を、企画者は画面遷移やマネタイゼーション要素を考慮
    - 多角的な視点により、後工程でのトラブルを予防できる

- <span style ="color: red;">**職種間理解の促進**</span>
    - 見積もりの過程で、各メンバーが持つ専門知識がチーム全体に共有される
    - 企画者は技術的制約を、エンジニアはユーザー体験の複雑さを理解できる
    - 相互理解により、より効率的なコミュニケーションが可能になる

- <span style ="color: red;">**コミットメントの向上**</span>
    - チーム全体で決めた見積もりに対して、メンバー全員が責任感を持ちやすくなる
    - 個人の押し付けではなく、合意に基づく計画となる

## **トランプを使ったポイント制度**

一般的なプランニングポーカーではフィボナッチ数列（1, 2, 3, 5, 8, 13, 21...）が使用されるが、  
今回はイメージしやすいトランプを使用する。

### **使用するカードとポイント対応表**

| カード | ポイント | 工数の目安 | 具体例（ゲーム開発） |
|--------|----------|------------|---------------------|
| A（エース）| 1 | 超簡単（0.5日以内） | 既存UIの文字修正、パラメーター調整 |
| 2 | 2 | 簡単（1日程度） | 新しいボタンの追加、簡単なエフェクト追加 |
| 3 | 3 | やや簡単（1.5日程度） | 既存機能の小規模修正、テキスト実装 |
| 5 | 5 | 普通（2-3日程度） | 新しい画面の実装、中程度のゲーム機能 |
| 8 | 8 | やや難しい（1週間程度） | 新しいゲームシステム、複雑なUI |
| K（キング）| 13 | 難しい（2週間程度） | 大きな機能追加、システム改修 |
| ジョーカー | ∞ | 不明・要分割 | 見積もり不可能、細分化が必要 |

### **ポイント選択の考え方**

<span style ="color: red;">**相対的な評価を重視**</span>  
絶対的な時間ではなく、他のタスクとの相対的な重さで判断する。  
例えば、「ログイン機能」を5ポイントとした場合、「アイテム管理機能」は  
それより簡単か複雑かを基準に考える。

<span style ="color: red;">**不確実性も考慮**</span>  
技術的な調査が必要な場合や、外部依存がある場合は、ポイントを高めに設定する。  
ゲーム開発では特に、「実装してみないとわからない」要素が多いため、  
この点は重要である。

## **実施チーム構成**

### **チーム人員内訳**

この講義では6名体制（企画役4名、エンジニア役2名）で実施。  
企画役は企画の目線で、エンジニア役はエンジニアの目線でロールプレイする。  

### **各チーム内での役割分担例**

**企画**

- **ディレクション役**：ゲームを面白くできるかどうかの目線で
- **企画作業者役**：その変更が企画的に適しているかどうかの目線で
- **ユーザー役**：開発に関わらない 1 ユーザーの目線で

**エンジニア**

- **実装者役**：実際に「やる」となった際の作業負荷の目線で
- **システム設計役**：他システムとの矛盾が無いかなどの設計の目線で

## **プランニングポーカーの実施手順**

### **事前準備（会議前に行うこと）**

1. **チーム編成**  
   参加者を 6 名に分割し、各チーム内で役割分担を決定

2. **見積もり対象の準備**  
   機能要件の一覧を作成し、  
   各項目は具体的で理解しやすい表現にする

3. **環境の準備**  
   各参加者にトランプ1組（A、2、3、5、8、K、ジョーカーのみ使用）、  
   意見の纏めの為、ホワイトボードまたは画面共有の環境、  
   タイマー（議論時間管理用）を用意

### **実施ステップ**

1. **タスクの説明（5分）**  
   司会者が見積もり対象のタスクを説明する

2. **チーム内議論（10分）**  
   6名 で議論を行い、情報や懸念等を洗い出す  
   この時、自分の見積もり（ポイント）を挙げたりしない様に

3. **見積もりの発表（5分）**  
   個々の見積もりを一斉に開示する

4. **全体議論（10分 x 繰り返し回数）**  
   チームで見積もりが大きく異なる場合、その理由を深掘りして議論する

5. **再見積もり（5分 x 繰り返し回数）**  
   議論を踏まえて再検討し、新しい見積もりを提示する

6. **最終合意**  
   全体で最終的な見積もりポイントを決定し、記録する

### **具体的なタスク説明例**

**例：「プレイヤーがゲーム内アイテムを他のプレイヤーとトレードできる機能を追加して」**

- 必要な画面：トレード申請、アイテム選択、確認画面
- 技術要件：サーバー連携、データ同期、セキュリティ
- 企画要件：手数料システム、レート表示、不正防止

### **チーム内議論の例**

**チーム内の議論例**

- Aさん：「UI設計と仕様策定が必要…！」
- Bさん：「ユーザーテストと調整も必要なので、より工数かかりそう…」  
- Cさん：「UI は既存のアイテム画面流用でいけると思う」
- Dさん：「データベース設計から必要だから結構掛かる…」
- Eさん：「既存のアイテム管理 API があるはずなのでその拡張だけで良くない？」
- Fさん：「セキュリティ対策を考慮するときつい」

## **ハンズオンを始める前に**

### **企画者向けで意識しておく事**

<span style ="color: red;">**技術的制約の理解**</span>  
「なぜこの機能がそんなに工数がかかるの？」から  
「技術的に複雑な理由がある」への理解変化を目指す。

<span style ="color: red;">**具体的な要件定義スキル**</span>  
曖昧な仕様（「かっこいいエフェクト」「使いやすいUI」）が、  
どれだけエンジニアを困らせ、工数を増大させるかを体験的に学習する。

<span style ="color: red;">**現実的な交渉スキル**</span>  
技術者との建設的な議論方法、妥協点の見つけ方、  
段階的な実装計画の立案方法を習得する。

### **エンジニア向けで意識しておく事**

<span style ="color: red;">**非技術者への説明スキル**</span>  
技術的複雑さを企画者に分かりやすく伝える能力を向上させる。  
「データベースが複雑だから」ではなく「なぜ複雑で、どう影響するか」を  
説明できるようになる。

<span style ="color: red;">**ビジネス視点の理解**</span>  
企画要件の背景にあるユーザー体験やビジネス目標を理解し、  
技術的な実装方針に反映できるようになる。

<span style ="color: red;">**見積もり根拠の整理**</span>  
「なんとなく大変そう」から「具体的な作業項目とその理由」へ、  
見積もりの精度と説得力を向上させる。

## **ハンズオンにおける注意点とコツ**

### **進行のポイント**

<span style ="color: red;">**時間管理の徹底**</span>  
6名体制では議論が活発になりがちである。各フェーズの時間を厳格に管理し、  
以下のタイムテーブルを守る

- タスク説明：3分
- チーム内議論：5分
- 発表・質疑：5分
- 全体議論：10分（必要に応じて）
- 再見積もり：3分

<span style ="color: red;">**建設的な議論の促進**</span>  
職種別に分かれることで対立構造が生まれやすくなる可能性がある。  
「企画 vs エンジニア」ではなく「チーム全体でより良い見積もりを作る」という  
共通目標を強調する。

<span style ="color: red;">**中立性の維持**</span>  
司会者は特定の職種に偏った発言を避け、両チームから均等に意見を引き出すよう心がける。

### **チーム内合意のコツ**

**企画としての効果的な議論**

- ユーザー体験の複雑さ
- 競合他社との差別化要素
- マネタイゼーションへの影響
- 運営・アップデート時の工数

**エンジニアとしての効果的な議論**

- 既存システムとの連携度
- 新技術導入の必要性
- パフォーマンスへの影響
- 保守性・拡張性の考慮

### **よくある失敗パターンと対策**

- <span style ="color: red;">**チーム内で強い意見に流される**</span>
    - 3名の中で経験豊富な人の意見に他の2名が無条件で従ってしまう
    - 対策：司会者が各チーム内の議論プロセスも観察し、全員の意見を聞くよう促す

- <span style ="color: red;">**職種間の対立激化**</span>
    - 企画とエンジニアで見積もりが大きく異なり、感情的な対立に発展する
    - 対策：「なぜそう思うか」の理由に焦点を当て、人格攻撃を避ける

- <span style ="color: red;">**妥協による安易な中間値採用**</span>
    - 議論を避けて単純に平均値を取ってしまう
    - 対策：合意の根拠を明確化し、記録に残す

## **ハンズオン課題**

### **課題1：パズルゲームのヒント機能追加**

**背景**  

シンプルなパズルゲーム（テトリスのようなもの）で、  
プレイヤーから「難しすぎて詰まってしまう」という声が多く寄せられている。  
初心者向けにヒント機能を追加することになった。

**要件**

- 「ヒント」ボタンを画面に追加
- ボタンを押すと、次に置くべきブロックの最適な位置を光らせる
- 1ゲームにつき3回まで使用可能
- ヒント使用回数の表示
- 既存のゲーム進行には影響なし

**企画側の検討ポイント**

- ヒントの分かりやすさ（光り方、色など）
- 使用回数制限は適切か
- ゲームバランスへの影響は？

**エンジニア側の検討ポイント**

- 最適解の計算方法
- 既存コードへの影響範囲
- UIボタンの実装

---

### **課題2：RPGの新モンスター追加**

**背景**  

既存のRPGゲームに新しいモンスターを5種類追加する。  
プレイヤーの飽きを防ぐため、定期的に新モンスターを追加することが決まった。

**要件**

- 新モンスター5種類（スライム系、ゴブリン系など）
- それぞれ異なる攻撃パターン（3パターン程度）
- HP、攻撃力、経験値などのパラメーター設定
- ドロップアイテムの設定
- 既存のダンジョンに配置

**企画側の検討ポイント**

- 既存モンスターとのバランス調整
- ドロップアイテムの経済バランス
- プレイヤーレベルに応じた配置場所

**エンジニア側の検討ポイント**

- 既存のモンスターシステムの活用
- 新しい攻撃パターンの実装
- データ設定の工数

---

### **課題3：アクションゲームのリプレイ機能**

**背景**  

アクションゲームで、プレイヤーが自分のプレイを録画・再生できる  
リプレイ機能を追加する。上手なプレイを友達に見せたり、  
失敗を振り返ったりできるようになる。

**要件**

- ゲームプレイの録画機能
- 録画したデータの保存（本体に3つまで）
- リプレイの再生機能
- 再生中の一時停止、早送り
- リプレイデータの削除機能

**企画側の検討ポイント**

- どのくらいの長さまで録画するか
- ファイルサイズの制限
- 操作の分かりやすさ

**エンジニア側の検討ポイント**

- データの記録方式（動画？操作ログ？）
- ファイル圧縮の必要性
- 再生時のパフォーマンス

---

### **課題4：レーシングゲームのゴースト機能**

**背景**  

レーシングゲームで、自分の過去のベストタイムと一緒に走れる  
「ゴースト機能」を実装する。プレイヤーが自分の記録更新により  
挑戦しやすくなる。

**要件**

- 各コースのベストタイムを自動記録
- ゴーストカー（半透明の車）の表示
- ゴースト表示のON/OFF切り替え
- ゴーストとの差タイム表示
- 記録更新時の演出

**企画側の検討ポイント**

- ゴーストの見た目（どのくらい透明に？）
- 差タイム表示のタイミング
- 記録更新の盛り上げ方

**エンジニア側の検討ポイント**

- 車の位置データの記録方法
- ゴースト描画の技術的課題
- データ保存の仕組み

## **効果的な工数予測に向けて**

プランニングポーカーは単なる見積もり手法ではなく、  
<span style ="color: red;">**チーム全体のコミュニケーション改善ツール**</span>。  
特にゲーム開発において、創造性と計画性のバランスを取りながら  
プロジェクトを成功に導くために欠かせないスキルといえる。

### **実践への第一歩**

今回の講義で学んだプロセスを、まずは小さなタスクから  
実際のプロジェクトに適用してみること。  
完璧を求めず、継続的な改善を心がけることで、  
チーム全体の見積もりスキルが向上し、より良いゲーム作りに貢献できる。

### **継続的なスキル向上**

<span style ="color: red;">**見積もり精度の向上**</span>  
過去の見積もりと実績を比較分析し、チーム固有の傾向を把握する。  
「企画チームはUI作業を過小評価しがち」「エンジニアチームは新技術を過大評価しがち」  
といったパターンを発見できる。

<span style ="color: red;">**コミュニケーション品質の向上**</span>  
職種間の専門用語の理解が進み、より効率的で正確な議論ができるようになる。

<span style ="color: red;">**プロジェクト管理スキルの向上**</span>  
工数見積もりの精度向上により、より現実的で実行可能なスケジュールを立案できるようになる。

工数予測の精度向上は、プロジェクトの成功確率を大幅に高める。  
今回プランニングポーカーという手法で体験してもらったが、このやり方自体が重要なわけではなく、  
あくまで工数計算精度を上げる事が重要だという事を覚えておいて欲しい。
