# **Windowsプログラミング**

DirectX12 を実際に触る前に、Windowsアプリケーションとしてのゲームプログラムの基礎を用意する必要がある。  
物を描画する為の空のウィンドウを用意してから、でなければ、物を描画する事は出来ない。

---

## **Win32 API**

Win32 API　はプログラムと Windows OS が会話するための「共通語」のようなもの。  
プログラムが Windows に「ウィンドウを作ってください」「ファイルを開いてください」などの、  
様々な依頼を行う為の専用の機能の纏まりを指す。

<span style="color: red;">**全ての Windows アプリケーション（ブラウザ、ゲーム、メモ帳など）は Win32 API を使用している**</span>

普段使っているアプリケーションも、  
内部では無数の Win32 API 関数を呼び出して Windows と連携している。  
ゲーム開発でもこの基礎的な仕組みは把握しておかないといけない。


## **処理フロー**

大まかな処理の流れは下記の通り。

1. **アプリケーション起動** → WinMain関数実行
2. **ウィンドウクラス登録** → 「こんなウィンドウを作りたい」をWindowsに登録
3. **ウィンドウ作成** → 実際にウィンドウを表示
4. **メッセージループ** → ユーザーの操作を待機
5. **メッセージ受信** → ウィンドウプロシージャで処理
6. **アプリケーション終了** → メッセージループ終了

この基本的な流れを理解することで、  
どんな Windows アプリケーションでも作成できるようになる。

## **アプリケーション起動**

通常のC++プログラムは`main()`関数から実行が開始されるが、  
Windows アプリケーションでは`WinMain()`関数が実行の起点となる。  
これは「Windowsアプリケーション専用の入り口」だと考えて問題ない。  

コンソールアプリケーションと違い、ウィンドウを持つアプリケーションは、  
Windows から様々な情報を受け取って動作する必要がある。

```cpp
int WINAPI WinMain(
    HINSTANCE hInstance,      // アプリケーションの識別番号
    HINSTANCE hPrevInstance,  // 基本使わなくていい
    LPSTR lpCmdLine,          // コマンドライン引数（起動時のオプション）
    int nCmdShow              // ウィンドウの表示方法（最大化、最小化など）
)
{
    // ここにメインの処理を書く
    // 1. ウィンドウクラス登録
    // 2. ウィンドウ作成
    // 3. メッセージループ
    return 0;
}
```

## **ウィンドウクラス登録**

ウィンドウを作る前に「どんなウィンドウを作りたいか」を Windows に教える必要がある。  
これがウィンドウクラスの登録となる。

```cpp
WNDCLASS wc{};
wc.lpfnWndProc = WindowProc;           // ウィンドウプロシージャを指定
wc.hInstance = hInstance;              // アプリケーションインスタンス
wc.lpszClassName = L"GameWindow";      // ウィンドウクラス名
wc.hCursor = LoadCursor(NULL, IDC_ARROW);  // マウスカーソル
wc.hbrBackground = (HBRUSH)(COLOR_WINDOW + 1);  // 背景色

RegisterClass(&wc);  // Windowsに登録
```

## **ウィンドウ生成**

登録したウィンドウクラスを使って実際のウィンドウを作成。

```cpp
HWND hwnd = CreateWindow(
    L"GameWindow",        // ウィンドウクラス名
    L"My Game",           // ウィンドウタイトル
    WS_OVERLAPPEDWINDOW,  // ウィンドウスタイル
    CW_USEDEFAULT, CW_USEDEFAULT,  // 位置（自動）
    800, 600,            // サイズ（幅、高さ）
    NULL, NULL,          // 親ウィンドウ、メニュー
    hInstance,           // インスタンス
    NULL                 // 追加データ
);

ShowWindow(hwnd, nCmdShow);  // ウィンドウを表示
```

## **メッセージループ**

Windows は**イベントドリブン**という方式で動作している。  
「何かが起こるまで待機し、何か起こったら対応する」という仕組み。  

今回の場合、上記の「何か」は**メッセージ受信**の事を指している。

Windows から送られてくるメッセージを常に監視し、  
メッセージが届いたら適切な処理に受け渡す必要がある。

その役割を**メッセージループ**が担っている。

```cpp
MSG msg{};  // メッセージを格納する構造体
while (GetMessage(&msg, NULL, 0, 0))  // メッセージが来るまで待機
{
    TranslateMessage(&msg);  // キーボードメッセージを使いやすい形に変換
    DispatchMessage(&msg);   // 適切なウィンドウプロシージャに送信
}
```

上記の例はメッセージが来るまで待機し続けるが、  
<span style="color: red;">**メッセージが止まると、アプリケーション全体が反応しなくなる**</span>。  
実際のゲームアプリケーションでは、メッセージが無くてもループの進行を止めてはならない。  
その為`GetMessage`はゲームアプリケーションでの利用に向いていない。

### **主なメッセージの種類**

- **WM_PAINT**：画面の再描画要求
- **WM_KEYDOWN/WM_KEYUP**：キーボード入力
- **WM_LBUTTONDOWN**：左クリック
- **WM_SIZE**：ウィンドウサイズ変更
- **WM_CLOSE**：ウィンドウを閉じる要求

## **メッセージ受信**

受け取ったメッセージは**ウィンドウプロシージャ**に受け渡される。  

ウィンドウプロシージャは受付係のような存在。  
マウスクリック、キーボード入力、ウィンドウサイズ変更、アプリケーション終了要求など、  
ウィンドウに起こるすべての出来事を受け取り適切に処理する。

```cpp
LRESULT CALLBACK WindowProc(HWND hwnd, UINT uMsg, WPARAM wParam, LPARAM lParam)
{
    switch (uMsg)
    {
    case WM_DESTROY:  // ×ボタンが押された時
        PostQuitMessage(0);  // 「アプリを終了します」とWindowsに通知
        return 0;
    case WM_PAINT:    // ウィンドウの再描画が必要な時
        // 画面を更新する処理をここに書く
        return 0;
    case WM_KEYDOWN:  // キーが押された時
        // キー入力の処理をここに書く
        return 0;
    }
    // 自分で処理しないメッセージはWindowsに任せる
    return DefWindowProc(hwnd, uMsg, wParam, lParam);
}
```

様々な「出来事（イベント）」に対する処理を定義することで、  
インタラクティブなアプリケーションが作成できる。
