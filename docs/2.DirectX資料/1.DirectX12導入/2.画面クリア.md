# **画面クリア**

---

## **画面クリア処理の全体像**

DirectX12における画面クリア処理は、複数の機能が連携して動作する複雑なシステムである。  
初期化、コマンドキュー、スワップチェーンに続いて、実際に画面に描画を行うためには、  
コマンドの記録と実行、GPU 同期処理が必要となる。

### **画面クリア処理に必要な機能と流れ**

画面をクリアするまでに必要な機能。

- <span style="color: red;">**ディスクリプタヒープ**</span>：GPU側でリソースにアクセスするための参照情報を格納
- <span style="color: red;">**レンダーターゲット**</span>：描画結果を出力する先のリソース
- <span style="color: red;">**コマンドアロケータ**</span>：コマンドリストが使用するメモリを管理
- <span style="color: red;">**コマンドリスト**</span>：GPUに送信する描画コマンドを記録
- <span style="color: red;">**フェンス**</span>：CPU-GPU間の同期処理を制御

画面をクリアするまでの大枠の流れ。

1. **各機能を作成** → クリアまでに必要な各機能を作成
2. **コマンドリストのリセット** → GPU にやらせたい事を記録するノートをリセット
3. **コマンドリストに各処理を記録** → GPU にやらせたい事を順番に記録する
4. **コマンドリストを実行** → 記録した内容の実行を開始する
5. **フェンス値を更新** → GPU にやらせている事が終わったら教えてもらえるようにする
6. **スワップチェーンでプレゼント** → GPUに描かせた絵を画面に表示
7. **フェンス値を元に GPU と CPU を同期** → 今描かせている絵を上書きさせない様に待機

---

## **ディスクリプタヒープ**

ディスクリプタヒープは、GPUがリソースにアクセスするための「目次」のような役割を果たす。  
DirectX12では、テクスチャやバッファなどのリソースに直接アクセスするのではなく、  
ディスクリプタという参照情報を介してアクセスを行う。

### **ディスクリプタヒープの特徴**

ディスクリプタヒープには以下の特徴がある。

- **連続したメモリ領域**：複数のディスクリプタが連続して配置される
- **型別管理**：RTV（レンダーターゲット）、DSV（デプス）、CBV/SRV/UAV（シェーダリソース）など
- **GPUメモリ上に配置**：高速なアクセスを実現するためGPU側で管理

### **ディスクリプタヒープの作成例**

```cpp
// RTVディスクリプタヒープの作成例
D3D12_DESCRIPTOR_HEAP_DESC rtvHeapDesc = {};
rtvHeapDesc.NumDescriptors = 2;  // スワップチェーンのバッファ数
rtvHeapDesc.Type = D3D12_DESCRIPTOR_HEAP_TYPE_RTV;
rtvHeapDesc.Flags = D3D12_DESCRIPTOR_HEAP_FLAG_NONE;

ID3D12DescriptorHeap* rtvHeap;
device->CreateDescriptorHeap(&rtvHeapDesc, IID_PPV_ARGS(&rtvHeap));
```

### **ディスクリプタハンドルによるアクセス**

ディスクリプタヒープ内の特定の位置にアクセスするため、ディスクリプタハンドルを使用する。

```cpp
// RTVハンドルの取得例
D3D12_CPU_DESCRIPTOR_HANDLE rtvHandle = rtvHeap->GetCPUDescriptorHandleForHeapStart();
UINT rtvDescriptorSize = device->GetDescriptorHandleIncrementSize(D3D12_DESCRIPTOR_HEAP_TYPE_RTV);

// 2番目のRTVハンドルの計算
rtvHandle.ptr += rtvDescriptorSize;
```

## **レンダーターゲット**

レンダーターゲットは描画結果を出力する先のリソースの事を指す。  
GPU に絵を描かせる為の画用紙のような物と考えればいい。  

### **レンダーターゲットの作成例**

スワップチェーンで用意したバッファをレンダーターゲットとして使用する。  
スワップチェーン作成時、バッファは二つ用意されている為、レンダーターゲットも二つ用意する。

当然スワップチェーンのバッファを利用せず、独立したレンダーターゲットを作成する事も可能。

```cpp
// レンダーターゲット作成例
ID3D12Resource* renderTargets[2];  // スワップチェーンでバッファが二つあるのでダブルバッファで

for (int i = 0; i < 2; i++)
{
    // スワップチェーンからバッファを取得
    swapChain->GetBuffer(i, IID_PPV_ARGS(&renderTargets[i]));
    
    // RTVハンドルの計算
    D3D12_CPU_DESCRIPTOR_HANDLE rtvHandle = rtvHeap->GetCPUDescriptorHandleForHeapStart();
    rtvHandle.ptr += i * rtvDescriptorSize;
    
    // RTVの作成
    device->CreateRenderTargetView(renderTargets[i], nullptr, rtvHandle);
}
```

### **バッファ切替の仕組み**

ダブルバッファリングでは、フロントバッファとバックバッファを交互に切り替える。
この切替はスワップチェーンを介して行われる。

- <span style="color: red;">**フロントバッファ**</span>：現在画面に表示されているバッファ
- <span style="color: red;">**バックバッファ**</span>：次フレームの描画を行うバッファ

## **コマンドアロケータ**

コマンドアロケータは、コマンドリストが使用するメモリを管理するオブジェクト。  
GPU に行わせる仕事で必要な道具は、アロケータから提供されるイメージ。  

### **アロケータを複数用意する理由**

<span style="color: red;">**重要な設計原則**</span>として、  
**実行中のコマンドが使用しているアロケータはリセットしてはならない。**  
そのため、バッファごとに専用のアロケータを用意する。

後述のフェンス処理にも関係する。

### **コマンドアロケータの作成例**

```cpp
// コマンドアロケータ作成例
ID3D12CommandAllocator* commandAllocators[2];

for (int i = 0; i < 2; i++)
{
    device->CreateCommandAllocator(
        D3D12_COMMAND_LIST_TYPE_DIRECT,
        IID_PPV_ARGS(&commandAllocators[i])
    );
}
```

### **アロケータの使い分け**

現在のバックバッファインデックスに対応するアロケータを使用する。

| バッファインデックス | 使用するアロケータ | 状態 |
|---------------------|------------------|------|
| 0 | commandAllocators[0] | 描画コマンド記録中 |
| 1 | commandAllocators[1] | GPU実行中（リセット不可） |

---

## **コマンドリスト**

コマンドリストは、GPU に送信する描画コマンドを記録するオブジェクトである。  
記録されたコマンドは、後でコマンドキューを通じてGPUに送信される。

コマンドリストはコマンドアロケータと違い 1 つでいい。  
コマンドリストは「やってほしい事の記録メモ」であり、  
記録メモの内容を GPU に渡せた後なら、  
現在のメモを消して、新しい記録を書き込んでいっても問題ない。

### **コマンドリストの基本的な使用手順**

コマンドリストの使用は以下の手順で行う。

1. **リセット**：前回の記録内容をクリア & 利用するコマンドアロケータを設定
2. **記録**：描画コマンドを順次記録
3. **クローズ**：記録終了を明示
4. **実行**：コマンドキューに送信

### **コマンドリストの作成例**

```cpp
// コマンドリスト作成例
ID3D12GraphicsCommandList* commandList;
device->CreateCommandList(
    0,
    D3D12_COMMAND_LIST_TYPE_DIRECT,
    commandAllocators[currentBackBuffer],
    nullptr,
    IID_PPV_ARGS(&commandList)
);

// 初期状態でクローズしておく
commandList->Close();
```

### **画面クリアコマンドの記録**

実際の画面クリア処理では、以下のコマンドを記録して実行する。

```cpp
// 現在のバックバッファインデックスを取得
UINT currentBackBuffer = swapChain->GetCurrentBackBufferIndex();

// コマンドリストをリセット
commandAllocators[currentBackBuffer]->Reset();
commandList->Reset(commandAllocators[currentBackBuffer], nullptr);

// レンダーターゲットを「絵を描く状態」に変更する（リソースバリア）
D3D12_RESOURCE_BARRIER barrier = {};
barrier.Type = D3D12_RESOURCE_BARRIER_TYPE_TRANSITION;
barrier.Transition.pResource = renderTargets[currentBackBuffer];
barrier.Transition.StateBefore = D3D12_RESOURCE_STATE_PRESENT;
barrier.Transition.StateAfter = D3D12_RESOURCE_STATE_RENDER_TARGET;
commandList->ResourceBarrier(1, &barrier);

// レンダーターゲットの取得
D3D12_CPU_DESCRIPTOR_HANDLE rtvHandle = rtvHeap->GetCPUDescriptorHandleForHeapStart();
rtvHandle.ptr += currentBackBuffer * rtvDescriptorSize;

// 画面クリア（青色）
float clearColor[] = { 0.0f, 0.0f, 1.0f, 1.0f };
commandList->ClearRenderTargetView(rtvHandle, clearColor, 0, nullptr);

// レンダーターゲットを「絵を出す状態」に変更する（リソースバリア）
barrier.Transition.StateBefore = D3D12_RESOURCE_STATE_RENDER_TARGET;
barrier.Transition.StateAfter = D3D12_RESOURCE_STATE_PRESENT;
commandList->ResourceBarrier(1, &barrier);

// コマンドリストをクローズ
commandList->Close();


// ここまでやって「画面をクリアする」のコマンドリストが完成する


```

### **画面クリアコマンドの実行**

コマンドリストの記録が終わったらキューに実行させる。

```c++
// 「画面をクリアする」コマンドリストを実行 → GPU が記録内容の実行を開始する
commandQueue->ExecuteCommandLists(1, (ID3D12CommandList**)&commandList);
```

## **フェンス**

フェンスは、CPU と GPU の処理を同期するためのオブジェクトである。  
DirectX12では明示的な同期制御が必要で、特にバッファ切替時の同期は重要。

今回の例に合わせて端的にいうと、  
- **コマンドアロケータが使われている最中に、リセットされるのを避ける**  
- **画用紙に絵を描かせている最中に、同じ画用紙に新しく別の絵を描かせようとするのを避ける**  

為に行う。

### **フェンスの基本概念**

フェンスは「カウンタ」として動作し、GPU が特定の処理を完了した時点で値が更新される。

- <span style="color: red;">**フェンス値**</span>：GPU処理の進行状況を表す数値
- <span style="color: red;">**完了値**</span>：CPUが期待するGPU処理の完了点
- <span style="color: red;">**待機処理**</span>：CPU側でGPU完了を待つ処理

### **フェンスの作成例**

```cpp
// フェンスの生成例
ID3D12Fence* fence;
UINT64 fenceValue = 0;
HANDLE fenceEvent;

// フェンスの生成
device->CreateFence(0, D3D12_FENCE_FLAG_NONE, IID_PPV_ARGS(&fence));
// フェンスイベントの生成
fenceEvent = CreateEvent(nullptr, FALSE, FALSE, nullptr);

```

### **フェンスの利用例**

```cpp
// GPU で行われているコマンドリストの処理が全て終わったら 1 になる様に設定する
fenceValue = 1;
commandQueue->Signal(fence, fenceValue);

// まだ 1 になっていない（GPU処理が終わっていない）
if (fence->GetCompletedValue() < fenceValue)
{
    // 1 になった時に fenceEvent を行わせる
    fence->SetEventOnCompletion(fenceValue, fenceEvent);
    // fenceEvent が行われるまで CPU を止める
    WaitForSingleObject(fenceEvent, INFINITE);
}
```

### **バッファ切替におけるフェンス処理の複雑さ**

<span style="color: red;">**重要な注意点**</span>として、バッファ切替時のフェンス処理は単純な待機処理では不十分。  
各バッファが独立した処理タイミングを持つため適切な同期制御が必要となる。

「画用紙に絵を描いている間に、別の画用紙に新しく絵を描く」のは OK だが、  
「画用紙に絵を描いている間に、同じ画用紙に新しく絵を描く」のは NG。  

それが回避出来るようにフェンス処理を行う必要がある。

#### **問題となるケース**

```cpp
// ❌ 間違った実装例
commandQueue->ExecuteCommandLists(1, (ID3D12CommandList**)&commandList);
commandQueue->Signal(fence, ++fenceValue);

// 毎フレーム待機（効率が悪い）
if (fence->GetCompletedValue() < fenceValue)
{
    fence->SetEventOnCompletion(fenceValue, fenceEvent);
    WaitForSingleObject(fenceEvent, INFINITE);
}
```
#### **正しいバッファ別フェンス管理**

各バッファに対して独立したフェンス値を管理する。

```cpp
// バッファごとのフェンス値を管理
UINT64 fenceValues[2] = { 0, 0 };
UINT currentBackBuffer;

// フレーム処理関数
void RenderFrame()
{
    currentBackBuffer = swapChain->GetCurrentBackBufferIndex();
    
    // 現在のバッファのGPU処理完了を待機
    UINT64 completedValue = fence->GetCompletedValue();
    if (completedValue < fenceValues[currentBackBuffer])
    {
        fence->SetEventOnCompletion(fenceValues[currentBackBuffer], fenceEvent);
        WaitForSingleObject(fenceEvent, INFINITE);
    }
    
    // コマンド記録・実行
    ・・・・
    
    // 現在のバッファの新しいフェンス値を設定
    fenceValues[currentBackBuffer] = ++fenceValue;
    commandQueue->Signal(fence, fenceValues[currentBackBuffer]);
    
    // スワップチェーン表示
    swapChain->Present(1, 0);
}
```

#### **具体的な動作の例**

ダブルバッファリングでの実際の動作を時系列で示す。

```
初期状態:
GetCompletedValue() = 0, fenceValues[0] = 0, fenceValues[1] = 0

フレーム1 (frameIndex=0):
前回使用チェック → スキップ（初回のため）
描画処理 → Signal(fence, 1) → fenceValues[0] = 1

フレーム2 (frameIndex=1):  
前回使用チェック → スキップ（初回のため）
描画処理 → Signal(fence, 2) → fenceValues[1] = 2

フレーム3 (frameIndex=0, バッファー0を再利用):
前回使用チェック → GetCompletedValue() < 1 ?
- GPUが高速: 2 < 1 → false（待機なし）
- GPUが低速: 0 < 1 → true（待機発生）
描画処理 → Signal(fence, 3) → fenceValues[0] = 3
```

<span style ="color: red;">**GPU が低速な場合に 3 フレーム目で初めて実際の同期処理が動作する**</span>

#### **CommandQueueの順序保証**

<span style ="color: red;">**同一CommandQueue内では必ず投入順序で実行される**</span>

処理時間が異なっても順序は維持される。

- フレーム1（軽い処理）→ Signal(1) → フレーム2（重い処理）→ Signal(2)
- GetCompletedValue()は必ず 0→1→2→3... の順序で変化
- 値が飛ぶことはない（0→2にはならない）

これにより、フェンス値による確実な同期処理が実現できる。


### **フェンス処理の詳細解説**

上記の実装における重要なポイントを詳しく説明する。

- <span style="color: red;">**待機タイミング**</span>

    - フレーム開始時に、使用予定のバッファのGPU処理完了を確認
    - 完了していない場合のみ待機処理を実行
    - これにより、不要な待機時間を削減
    
- <span style="color: red;">**フェンス値の更新**</span>

    - コマンド実行後、該当バッファのフェンス値を更新
    - グローバルなfenceValueをインクリメントして一意性を保証
    - バッファごとに最新の処理完了点を記録

- <span style="color: red;">**同期の最適化**</span>

    - バッファが2つの場合、理想的には1つが描画中、もう1つが表示中
    - 適切な同期により、CPUとGPUの並列処理を最大化

- <span style="color: red;">**フェンスはあくまで「CPUとGPU」の同期を行う**</span>

    - GPU 同士の同期はまた別の話
    - CommandQueue の Wait を利用しなければならない（割愛）

## **まとめ**

DirectX12における画面クリア処理は、複数の機能が協調して動作する複雑なシステム。  
特に重要なポイントは以下の通り。

- <span style="color: red;">**ディスクリプタヒープ**</span>：GPUリソースへの効率的なアクセス手段を提供
- <span style="color: red;">**レンダーターゲット**</span>：スワップチェーンバッファを描画先として設定
- <span style="color: red;">**コマンドアロケータ**</span>：バッファ数と同数を用意し、同期問題を回避
- <span style="color: red;">**コマンドリスト**</span>：GPU処理の詳細な制御を可能にする
- <span style="color: red;">**フェンス**</span>：CPU-GPU間の適切な同期制御を実現

これらの仕組みを理解し、適切に実装することで、効率的なDirectX12アプリケーションの基礎を構築できる。
