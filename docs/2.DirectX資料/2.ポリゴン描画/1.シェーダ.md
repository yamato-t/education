# **シェーダ**

## **シェーダとは**

シェーダは GPU 上で実行される小さなプログラムで、  
描画パイプラインの各段階で異なる処理を担当する。  
シェーダプログラムを独自にコーディングする事で、  
独特な描画表現を実現する事が出来る。

これまでコーディングしてきた C/C++/C# は CPU で動作するプログラムであり、  
GPU で動作するプログラムのコーディングには、別の言語を利用しなければならない。  
代表的なシェーダ言語として、HLSL(High-Level Shading Language) が挙げられる。

### **シェーダの種類と役割**

まず最初に触る事になる、最も基本的なシェーダは下記二つ。

| シェーダ種類 | 実行タイミング | 主な処理内容 |
|-------------|---------------|-------------|
| 頂点シェーダ | 各頂点に対して | 座標変換、ライティング計算 |
| ピクセルシェーダ | 各ピクセルに対して | 最終的な色の決定 |

## **まず覚えておくべき事**

### **頂点レイアウト**

頂点データをシェーダで利用する際に、  
その頂点データがどのようなフォーマットになっているかをシェーダに知らせる為に必要。  

```c++
// C++側

// 実際の頂点構造体
struct Vertex {
    float position[3];  // x, y, z で 12byte
    float color[4];     // r, g, b, a で 16byte
};

// 頂点レイアウト
D3D12_INPUT_ELEMENT_DESC layout[] = {
    {
        "POSITION",                                 // セマンティクス名(後述)
        0,                                          // セマンティクス毎の番号（後述）
        DXGI_FORMAT_R32G32B32_FLOAT,                // データ形式（position は 32bit x 3）
        0,                                          // 入力スロット(複数の頂点バッファを扱う際に意味が出てくる)
        0,                                          // バイトオフセット（頂点データの先頭から数えて何バイト目にあたるか）
        D3D12_INPUT_CLASSIFICATION_PER_VERTEX_DATA, // 「頂点毎のデータ」を表す
        0                                           // ここはゼロで固定
    },
    { "COLOR",    0, DXGI_FORMAT_R32G32B32A32_FLOAT, 0, 12, D3D12_INPUT_CLASSIFICATION_PER_VERTEX_DATA, 0 }
};
```

### **セマンティクス（Semantics）**

セマンティクスはシェーダの入力・出力データがどのような意味を持つかを  
GPU に伝えるためのラベル。  
後述のコードサンプルで、実際の指定箇所を確認できる。

シェーダコードでは「この情報は何を表す情報なのか」を明確にする必要があり  
それをセマンティクスで指定する。

頂点シェーダ、ピクセルシェーダそれぞれで定義されており様々な種類がある。  
代表的なセマンティクス例は下記の通り。

| セマンティクス | 利用する場所 | 主な用途 | 主なデータ型 |
|---------------|------|-------------|-----------|
| `POSITION` |頂点シェーダ |頂点座標（x,y,z） | float3 |
| `COLOR` | 頂点シェーダ、ピクセルシェーダ|頂点色 (r,g,b,a) | float4 |
| `TEXCOORD` | 頂点シェーダ、ピクセルシェーダ|テクスチャ座標 (u,v)| float2 |
| `NORMAL` | 頂点シェーダ|法線ベクトル（x,y,z） | float3 |
| `BINORMAL` |頂点シェーダ |従法線ベクトル（x,y,z） | float3 |
| `TANGENT` | 頂点シェーダ|タンジェントベクトル（x,y,z） | float3 |

なお補足として、主なデータ型はあくまで**用途**に照らし合わせているのみになる。  
**上記で挙げているセマンティクスは、実は全て float4 のデータ型までサポートしている。**  

### **システムセマンティクス（System Value Semantics）**

<span style ="color: red;">**SV_ で始まるセマンティクスはGPUが特別扱いする**</span>  
後述のコードサンプルで、実際の指定箇所を確認できる。

こちらにも様々な種類がある。  
代表的なものは下記の通り。

| セマンティクス | 利用する場所 | 用途 | データ型 |
|---------------|----------|------|---------|
| `SV_POSITION` | 頂点シェーダ（出力） | クリップ空間座標(x,y,z,w) | float4 |
| `SV_POSITION` | ピクセルシェーダ（入力） | スクリーン座標(x,z), デプス値(z) | float4 |
| `SV_TARGET` | ピクセルシェーダ | 最終描画色(r,g,b,a) | float1 ~ float4 |
| `SV_InstanceID` | 頂点シェーダ | 描画インスタンスのID | uint |
| `SV_VertexID` | 頂点シェーダ | 描画インスタンス毎の頂点のID | uint |
| `SV_PrimitiveID` | ジオメトリシェーダ、ピクセルシェーダ | 描画インスタンス毎のプリミティブ（ポリゴン）番号 | uint |
| `SV_DispatchThreadID` | コンピュートシェーダ | スレッドの通し番号 | uint3 |

これらを踏まえて改めてシェーダコードを確認する。

## **頂点シェーダ（Vertex Shader）**

各頂点に対して実行され、頂点の座標変換や属性の処理を行う。  
コードは前回と同じもの。  

描画する頂点の数だけ処理される。  
今回の例では 3 頂点で三角形ポリゴンを描画するので 3 回処理される。

今回のサンプルは非常に簡易な物である為、  
実は 3D 描画は一切行っていない。

```c
// VertexShader.hlsl

// 頂点シェーダの入力フォーマット（C++/CPU 側の頂点データ構造体と合う形になる）
struct VSInput
{
    float3 position : POSITION;  // 入力：頂点座標：POSITION セマンティクス
    float4 color    : COLOR;     // 入力：頂点色：COLOR セマンティクス
};

// 頂点シェーダの出力フォーマット
struct VSOutput
{
    float4 position : SV_POSITION; // 出力：変換後の座標：SV_POSITION セマンティクス
    float4 color    : COLOR;       // 出力：頂点色：COLOR セマンティクス
};

// 頂点シェーダの本体処理
VSOutput vs(VSInput input)
{
    // 出力フォーマット型の変数宣言
    VSOutput output;
    
    // 3D座標を同次座標に変換
    output.position = float4(input.position, 1.0f);
    
    // 色情報をそのまま次の段階に渡す
    output.color = input.color;
    
    return output;
}
```

### **頂点レイアウトと頂点シェーダ入力、セマンティクスの関係**  

CPU 側で用意する頂点データ、頂点シェーダ入力、指定しているセマンティクスは、  
ちゃんと関連性を持っている。

```c++
// C++側

// 頂点データ構造体
struct Vertex {
    float position[3];  // x, y, z で 12byte
    float color[4];     // r, g, b, a で 16byte
};

// 頂点レイアウト定義
D3D12_INPUT_ELEMENT_DESC layout[] = {
    { "POSITION", 0, DXGI_FORMAT_R32G32B32_FLOAT,    0, 0,  D3D12_INPUT_CLASSIFICATION_PER_VERTEX_DATA, 0 },
    { "COLOR",    0, DXGI_FORMAT_R32G32B32A32_FLOAT, 0, 12, D3D12_INPUT_CLASSIFICATION_PER_VERTEX_DATA, 0 }
};
```

```c
// hlsl 側

// 頂点構造体（C++ 側で定義された構造体やレイアウトと合わせる必要がある）
struct VSInput {
    float3 position : POSITION;  // C++側で float3
    float4 color    : COLOR;     // C++側で float4
};
```

## **ピクセルシェーダ（Pixel Shader）**

ラスタライゼーション後の各ピクセルに対して実行され、  
最終的な色を決定する。  
コードは前回と同じもの。

色を塗るピクセルの数だけ処理される。  
その為頂点シェーダと比べて、かなりの数の処理が行われる事になる。  
今回の例では「三角形の面積にあたるピクセルの個数」だけ処理される。  
画面の中で三角形が大きければ、それだけ多くの回数分処理される。


```c
// PixelShader.hlsl

// ピクセルシェーダの入力フォーマット（頂点シェーダの出力フォーマットと同じになるが内容は違う）
struct PSInput
{
    float4 position : SV_POSITION;  // 入力：スクリーン座標（ピクセル単位）：SV_POSITION セマンティクス
    float4 color    : COLOR;        // 入力：補間された色：COLOR セマンティクス
};

// ピクセルシェーダの本体処理
// 戻り値はピクセルに塗る色（float4型で SV_TARGET セマンティクス）
float4 ps(PSInput input) : SV_TARGET
{
    // 補間された頂点色をそのまま出力
    return input.color;
    
    // より複雑な処理の例：
    // return input.color * 0.8f;           // 暗くする
    // return float4(1.0f, 0.0f, 0.0f, 1.0f); // 全て赤色
}
```

## **同じセマンティクスを複数持つ場合**  

上記の各シェーダの入出力で指定しているセマンティクスに関して、  
同じセマンティクスを複数指定する事も可能だが、  
その場合は `XXXXX0`,`XXXXX1` といった形で末尾に番号を指定する。

頂点データで TEXCOORD を複数持つ場合は下記のように指定する。

```c++
// C++ 側

// 頂点構造体
struct Vertex {
    float uv0[2];
    float uv1[2];
};

// 頂点レイアウト定義
D3D12_INPUT_ELEMENT_DESC layout[] = {
    { "TEXCOORD", 0, DXGI_FORMAT_R32G32_FLOAT, 0, 0,  ... },
    { "TEXCOORD", 1, DXGI_FORMAT_R32G32_FLOAT, 0, 8,  ... },
    //            ↑ これがセマンティクス番号    
};
```
```c
// hlsl 側

// 頂点構造体
struct VSInput
{
    float2 uv0      : TEXCOORD0; // 末尾に番号を付ける
    float2 uv1      : TEXCOORD1;
};
```

### **複数レンダーターゲットに対して描画する場合**

今回のサンプルでは一つのレンダーターゲットに対してのみ描画しているが、  
**一度のピクセルシェーダで複数のレンダーターゲットに対して書き込む事も可能**。  
これを**Multiple Render Targets = MRTと呼ぶ**。  
MRTの場合、`SV_TARGET0`,`SV_TARGET1`といった形で末尾に番号を指定して構造体化し、  
構造体型を戻り値に指定する。  

```c
// hlsl 側

// ピクセルシェーダ出力構造体
struct PSOutput {
    float4 color0 : SV_TARGET0;
    float4 color1 : SV_TARGET1;
};

PSOutput PSMain(PSInput input)
{
    PSOutput o;
    o.color0 = float4(1, 0, 0, 1);
    o.color1 = float4(0, 1, 0, 1);
    return o;
}
```

但し MRT を行うためには、 C++/CPU 側でも、  
複数のレンダーターゲットを描画先に設定する処理が必要。

## **頂点シェーダからピクセルシェーダに至るまでの情報の変化**

各情報は頂点シェーダから始まり、ピクセルシェーダまで渡されていくが、    
**その間に様々な計算が手動 or 自動で行われている**。  

### **座標情報の変化**

#### **1. ローカル座標**
読み込んだアセットの頂点座標そのもの。  
頂点シェーダにはこの情報が頂点座標として入ってくる。

#### **2. ワールド座標**
「ゲーム世界の空間の何処にその頂点があるか」を表す。  
ローカル座標にワールド行列をかけ合わせる事で算出する。  
通常、頂点シェーダで計算する。  

#### **3. ビュー座標**
「カメラを基準にした空間の何処にその頂点があるか」を表す。  
ワールド座標にビュー行列（カメラ逆行列）をかけ合わせる事で算出する。  
通常、頂点シェーダで計算する。

#### **4. クリップ座標**
「画面内のどこにその頂点があるか」を表す。  
ビュー座標にプロジェクション行列をかけ合わせる事で算出する。  
頂点シェーダではこのクリップ座標まで計算し、次の工程に遷移する。

**補足：プロジェクション行列**  
ビュー座標を計算したのち、  
それが「画面内に入っているかどうか」を判定する必要がある。  
その判定で利用されるのが**プロジェクション行列**。  
プロジェクション行列には下記の二種類がある。

1. 透視投影（Perspective Projection）  
透視投影の計算を経る事で、画面上でパースを反映させた頂点位置を計算する事が出来る。  
    - 特徴　：遠くのものが小さく見える
    - 見え方：現実世界と同じ
    - 用途　：通常の 3D オブジェクト
2. 平行投影（Parallel Projection）/ 正射影（Orthographic Projection）  
平行投影の計算を経る事で、遠近を考慮しない頂点位置を計算する事が出来る。  
    - 特徴　：遠くても近くても同じサイズ
    - 見え方：距離による大きさの変化なし
    - 用途　：2D 描画、UI

プロジェクション行列でクリップ座標を計算した後、  
「 xyz それぞれが -w ~ w の範囲内に収まっているかどうか」で、  
画面内にいるかどうかを判断する事が出来るようになる。

#### **5. 正規化デバイス座標**
「クリップ座標を w で正規化した座標」を表す。  
クリップ座標の xyz の各要素を w で割る（透視除算）事で -1 ~ 1 の範囲の値に変換する。  
この計算は GPU が自動で行う。

#### **6. スクリーン座標**
「実際のピクセルの位置とデプス（深度）値」を表す。  
具体的には、クリップ座標から下記の計算（ビューポート変換）が行われた結果が入っている。  
```
x（スクリーンX）= (クリップ座標.x + 1) ÷ 2 × 画面幅
y（スクリーンY）= (1 - クリップ座標.y) ÷ 2 × 画面高さ
z（デプス）     = クリップ座標.z / クリップ座標.w
w              = 1 / クリップ座標.w
```
この計算は GPU が自動で行う。

この後にラスタライズが行われ、  
その後ピクセルシェーダに渡されている。

### **座標以外の情報の変化**

頂点シェーダが出力した座標以外、  
例えば  

- 色情報
- テクスチャ座標
- 法線

などの情報は**ラスタライザによって頂点間で線形補間される**。  
今回の例だとポリゴンの色がグラデーションで変化しているが、  
補間が効いているからこそ、このような結果になっている。

各情報は基本的に補間の対象になるが、  
下記に該当する場合は補間を行わない。

- 特定のシステムセマンティクスを指定している場合
    - `SV_POSITION` もその一つ
- 変数宣言でキーワードを指定している場合
    - `nointerpolation float4 color : COLOR;  // 補間を行わない`
    - 補間設定キーワードはピクセルシェーダ入力側の変数に指定する
    - 色の場合は単一色（先頭の頂点情報）で塗りつぶされる


### **情報遷移の纏め**

| 処理 | 処理の担当 |補足|
|---------------|-------------|-----------|
| 頂点座標の計算処理 |  頂点シェーダ | ローカル→ワールド→ビュー→クリップ変換 |
| クリッピング（視野外の描画物を除外）処理  | GPU 自動処理 |  |
| 透視除算処理 | GPU 自動処理 | クリップ→正規化デバイス変換 |
| ビューポート変換処理 | GPU 自動処理 | 正規化デバイス→スクリーン変換 |
| ラスタライズ | GPU 自動処理 | 補間対象の各情報をピクセル毎の値に補間する |
| ピクセル毎の色を計算 | ピクセルシェーダ | スクリーン座標やラスタライズ結果の補間値が入力される |
