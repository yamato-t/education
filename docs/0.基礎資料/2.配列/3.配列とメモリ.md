# **配列とメモリ**

---

## **確保しているメモリ領域を意識する**

配列で確保しているメモリ領域は、「要素数 × 型のバイト数」になる。  
当然ながら、要素数を超えたメモリ領域を更新してはならない。

```c++
// 要素数が 4 の int 型の配列を宣言する
int count[4];
count[0] = 15;
count[1] = 32;
count[2] = 11;
count[3] = 48;

// NG1
count[4] = 0;

// NG2
for(int i = 0; i <= 4; i++){
    count[i] = 0;
}

```

配列の要素数を超えたメモリ領域に<span style ="color: red;">**何の情報が保持されているかは分からない**</span>。  
その為、<span style ="color: red;">**このコードを書いた事で何が起こるかもわからない**</span>。

その上、厄介なのは<span style ="color: red;">**このコードがエラーにならない場合がある事**</span>。  
NG1 の例は環境依存になるかもしれないが、NG2 の例は大半でコンパイルエラーにならない。  
つまり<span style ="color: red;">**何が起こるか分からないコードを書いている事にすら気付けない**</span>。

こういった問題は<span style ="color: red;">**バッファオーバーラン**</span>と呼ばれ、  
必ず回避しなければならない物となる。

### **バッファオーバーランの厄介な点**

バッファオーバーランが特に厄介なのは、  
<span style ="color: red;">**不具合が発生した箇所と、原因となったコードが大きく離れる**</span>点にある。

配列の範囲外を書き換えた直後は、  
プログラムが正常に動いているように見える事が多い。  
実際に問題が表面化するのは、後から別の処理で  
その破壊されたメモリを読み込んだ瞬間となる。

仮に要素数を超えた領域が「プレイヤーキャラの体力」を保持している変数で確保されていた場合、  
for ループを呼び出すとなぜかプレイヤーが即死する意味不明な事態に陥る。

```c++
int count[4];
int hp = 100; // メモリ上で count の直後に配置される可能性がある

// 色々な処理

for(int i = 0; i <= 4; i++){
    count[i] = 0; // i が 4 の時に hp の領域を破壊している可能性がある
}

printf("%d\n", hp); // なぜか 0 が表示される
```

`count` と `hp` に直接の関係はないように見える。  
しかしメモリ上では隣接して配置される可能性があり、  
`count[4]` への書き込みが `hp` を上書きする。

結果として「なぜか hp が 0 になる」という症状だけが現れ、  
原因である `count[4]` とは全く別の場所で不具合が露見する。  

更にこういった問題は環境依存の影響が強く、  
<span style ="color: red;">**簡単に発生する場合もあれば、長期間発生しない場合もある**</span>。

実際の開発では変数の数が数十〜数百に及ぶため、  
どこで何が壊されているかを追うのに多大な時間がかかる。
