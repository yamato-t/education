    DirectX12 初期化

    DirectX12の全体構造

    DirectX12は複数のオブジェクトが階層的に連携する設計となっている。
    これらの関係性を映画制作現場に例えると理解しやすい。

    監督（Device）が全体を統括し、
    助監督（CommandQueue）が撮影指示を管理し、
    カメラマン（CommandList）が実際の撮影作業を実行する。
    そして映写技師（SwapChain）が完成した映像を観客（モニター）に届ける役割を担う。

    DirectX12オブジェクト階層

    オブジェクト 役割 映画制作での例 主な機能
    Device GPU制御の中核 映画監督 リソース作成・管理
    CommandQueue 命令実行管理 助監督 GPU作業の順序制御
    CommandList 具体的作業指示 撮影台本 描画命令の記録
    SwapChain 画面表示制御 映写技師 フレームバッファ管理

    これらのオブジェクトは相互依存関係にあり、正しい順序で初期化する必要がある

    DirectX12 初期化までの流れ

    下記の流れに沿って実装を進めていく。

    1. DXGI ファクトリー作成
    2. アダプター選択と取得
    3. DirectX12 デバイス作成
    4. コマンドキューの作成
    5. スワップチェーン作成

    おまけ 1. デバッグレイヤーの有効化

    DXGI ファクトリーの作成

    DXGIファクトリーは「GPU情報の窓口」になる。
    システムに搭載されたGPUの情報取得やスワップチェーンの作成を担当する。

    複数の GPU が搭載されたシステムでは、最適な GPU を選択する役割も果たす。

    IDXGIFactory4* CreateDXGIFactory()
{
    IDXGIFactory4* factory;
    UINT createFactoryFlags = 0;

#if defined(_DEBUG)
    // デバッグビルドではデバッグフラグを設定
    createFactoryFlags = DXGI_CREATE_FACTORY_DEBUG;
#endif

    HRESULT hr = CreateDXGIFactory2(createFactoryFlags, IID_PPV_ARGS(&factory));
    if (FAILED(hr))
    {
        // エラーハンドリング：ファクトリー作成失敗
        OutputDebugString(L"Failed to create DXGI Factory\n");
        return nullptr;
    }

    return factory;
}

    DXGI ファクトリーの主要機能

    DirectX による描画処理で頻繁に出てくるわけではないが、

    • GPU（アダプター）の列挙
    • ディスプレイモードの取得
    • スワップチェーンの作成
    • フルスクリーン状態の管理

    など「ディスプレイの情報」を取得しなければならない機会で必要になる。

    アダプター選択と取得

    アダプターとは、システム内の GPU を表すオブジェクトである。
    統合 GPU、専用 GPU、ソフトウェアレンダラーなど、複数の選択肢から最適なものを選ぶ必要がある。

    IDXGIAdapter* GetHardwareAdapter(IDXGIFactory4* factory)
{
    IDXGIAdapter* adapter;

    // ハードウェアアダプターを順番に検索
    for (UINT adapterIndex = 0; ; ++adapterIndex)
    {
        adapter = nullptr;
        if (DXGI_ERROR_NOT_FOUND == factory->EnumAdapters1(adapterIndex, &adapter))
        {
            break; // アダプターが見つからない場合は終了
        }

        // アダプター情報を取得
        DXGI_ADAPTER_DESC1 desc;
        adapter->GetDesc1(&desc);

        // ソフトウェアアダプターをスキップ
        if (desc.Flags & DXGI_ADAPTER_FLAG_SOFTWARE)
        {
            adapter->Release();
            continue;
        }

        // DirectX12対応テスト
        if (SUCCEEDED(D3D12CreateDevice(adapter, D3D_FEATURE_LEVEL_11_0, _uuidof(ID3D12Device), nullptr)))
        {
            return adapter; // 適切なアダプターを発見
        }

        adapter->Release();
    }

    return nullptr;
}

    アダプター選択の判断基準

    複数のグラフィックボードを持つ PC の場合は複数のアダプターが見つかる。
    その中からどれを選ぶのかは、

    • ハードウェア支援の有無
    • DirectX12機能レベルの対応状況
    • VRAM容量
    • 性能指標

    などの情報から検討する。

    DirectX12 デバイス作成

    デバイスは「GPU操作の司令塔」である。
    すべての DirectX12 リソース（テクスチャ、バッファ、シェーダー等）はこのデバイスを通じて作成される。

    DirectX を利用して何かをしたい場合はまずこれを作る事が必要。

    ID3D12Device* CreateD3D12Device(IDXGIAdapter1* adapter)
{
    ID3D12Device* device;

    // デバイス作成を試行
    HRESULT hr = D3D12CreateDevice(
        adapter,                    // 使用するアダプター
        D3D_FEATURE_LEVEL_11_0,     // 最小機能レベル
        IID_PPV_ARGS(&device)       // 作成されるデバイス
    );

    if (FAILED(hr))
    {
        // フォールバック：ソフトウェアアダプターを試行
        hr = D3D12CreateDevice(
            nullptr,                // 規定アダプター
            D3D_FEATURE_LEVEL_11_0,
            IID_PPV_ARGS(&device)
        );

        if (FAILED(hr))
        {
            OutputDebugString(L"Failed to create D3D12 Device\n");
            return nullptr;
        }

        OutputDebugString(L"Using software adapter (WARP)\n");
    }

    return device;
}

    デバイス作成時の検証項目：

    • 機能レベルの対応確認
    • 必要な機能の利用可能性チェック
    • メモリ制限の把握

    コマンドキューの作成

    コマンドキューは「GPU 作業の交通整理係」である。
    CPU で作成した描画命令を GPU に順序立てて送信し、並列実行を効率的に管理する。

    ID3D12CommandQueue* CreateCommandQueue(ID3D12Device* device)
{
    // コマンドキューの設定
    D3D12_COMMAND_QUEUE_DESC queueDesc = {};
    queueDesc.Type = D3D12_COMMAND_LIST_TYPE_DIRECT;    // 直接実行型
    queueDesc.Priority = D3D12_COMMAND_QUEUE_PRIORITY_NORMAL;  // 通常優先度
    queueDesc.Flags = D3D12_COMMAND_QUEUE_FLAG_NONE;    // 特別フラグなし
    queueDesc.NodeMask = 0;                             // 単一GPU使用

    ID3D12CommandQueue* commandQueue;
    HRESULT hr = device->CreateCommandQueue(&queueDesc, IID_PPV_ARGS(&commandQueue));

    if (FAILED(hr))
    {
        OutputDebugString(L"Failed to create Command Queue\n");
        return nullptr;
    }

    // デバッグ用の名前設定（任意だが推奨）
    commandQueue->SetName(L"Main Command Queue");

    return commandQueue;
}

    コマンドキューの種類と用途

    • DIRECT: 汎用的な用途（基本これを最初に使う機会が多い）
    • COMPUTE: 計算処理専用
    • COPY: データ転送専用

    「とりあえずポリゴンを描画する」を目標にしている為、
    まずは DIRECT でコマンドキューを作成しておく。

    スワップチェーン作成

    スワップチェーンは「画面表示の管制塔」になる。
    GPUで描画した結果を画面に表示するためのバッファ管理を行う。

    描画先のバッファと表示中のバッファを分ける事で、
    描画中の中途半端な状態が画面に反映されてしまう問題（ティアリングなど）を防ぐ。

    IDXGISwapChain3* CreateSwapChain(IDXGIFactory4* factory, ID3D12CommandQueue* commandQueue, HWND hwnd)
{
    // スワップチェーンの詳細設定
    DXGI_SWAP_CHAIN_DESC1 swapChainDesc = {};
    swapChainDesc.BufferCount = 2;                      // ダブルバッファリング
    swapChainDesc.Width = 1280;                         // 画面幅
    swapChainDesc.Height = 720;                         // 画面高
    swapChainDesc.Format = DXGI_FORMAT_R8G8B8A8_UNORM; // ピクセルフォーマット
    swapChainDesc.BufferUsage = DXGI_USAGE_RENDER_TARGET_OUTPUT;  // レンダーターゲット用
    swapChainDesc.SwapEffect = DXGI_SWAP_EFFECT_FLIP_DISCARD;     // 高速切替
    swapChainDesc.SampleDesc.Count = 1;                 // マルチサンプリングなし

    IDXGISwapChain1* swapChain1;
    HRESULT hr = factory->CreateSwapChainForHwnd(
        commandQueue,       // コマンドキュー
        hwnd,              // ターゲットウィンドウ
        &swapChainDesc,    // 設定
        nullptr,           // フルスクリーン設定
        nullptr,           // 出力制限
        &swapChain1        // 作成されるスワップチェーン
    );

    if (FAILED(hr))
    {
        OutputDebugString(L"Failed to create Swap Chain\n");
        return nullptr;
    }

    // より高機能なインターフェースにキャスト
    IDXGISwapChain3* swapChain;
    hr = swapChain1->QueryInterface(IID_PPV_ARGS(&swapChain));
    swapChain1->Release();

    if (FAILED(hr))
    {
        OutputDebugString(L"Failed to cast to SwapChain3\n");
        return nullptr;
    }

    return swapChain;
}

    スワップチェーン設定の重要パラメータ

    • BufferCount: バッファ数（2または3を推奨）
    • Format: 色深度とアルファチャンネル
    • SwapEffect: バッファ切替方式
    • SampleDesc: アンチエイリアシング設定

    オブジェクトの解放（Release）

    COM オブジェクトとは

    DirectX12 で作成する ID3D12Device、IDXGISwapChain、IDXGIAdapter、
    ID3D12CommandQueue、ID3D12Resource など、
    ID3D や IDXGI で始まる全てのオブジェクトは COM オブジェクトと呼ばれる

    COM（Component Object Model）とは、Windows が採用しているオブジェクトの仕組みで、
    異なる言語・環境からでも統一された方法でオブジェクトを扱えるようにするための規格。
    DirectX のオブジェクトもこの仕組みに則って設計されている。

    COM オブジェクトは内部で参照カウントを持っており、
    使い終わった際に Release() を呼んで参照カウントを減らす必要がある。
    参照カウントがゼロになった時点でオブジェクトが破棄され、メモリが解放される。

    Release() を呼ばずにポインタが消えると、メモリリークが発生する。
    作成した COM オブジェクトは必ず Release すること。

    // 使い終わったオブジェクトを解放する
// Release() を呼ぶと参照カウントが減り、0 になった時点でメモリが解放される
pipelineState->Release();
rootSignature->Release();
commandList->Release();
commandAllocator->Release();
rtvHeap->Release();
swapChain->Release();
commandQueue->Release();
device->Release();

    解放の順番

    作成した順序と逆順に Release するのが基本原則。
    依存関係のあるオブジェクトを先に解放しないと、内部で参照が残り正しく破棄されない場合がある。

    // 作成順：device → commandQueue → swapChain → ... → pipelineState
// 解放順：pipelineState → ... → swapChain → commandQueue → device

    デバッグレイヤーで確認する

    解放漏れがあると、デバッグビルドの終了時にデバッグレイヤーが
    以下のような警告を出力ウィンドウに表示する。

    D3D12 WARNING: Live ID3D12Device at 0x... with 1 ref counts.

    この警告が出た場合は、該当オブジェクトの Release() 漏れを疑うこと。

    ComPtr による自動解放

    Release() の呼び忘れを防ぐ仕組みとして、ComPtr が用意されている。
    ComPtr は COM オブジェクト専用のスマートポインタで、
    スコープを抜けた際に自動で Release() を呼んでくれる。
    #include <wrl/client.h> で利用でき、Microsoft::WRL::ComPtr<T> として使用する。

    #include <wrl/client.h>
using Microsoft::WRL::ComPtr;

// ComPtr で宣言すると、スコープを抜けた時に自動で Release される
ComPtr<ID3D12Device> device;
ComPtr<ID3D12CommandQueue> commandQueue;

// 作成方法は生ポインタと同じ（IID_PPV_ARGS に渡せる）
D3D12CreateDevice(adapter, D3D_FEATURE_LEVEL_11_0, IID_PPV_ARGS(&device));

// 生ポインタとして取り出す場合は .Get() を使う
someFunction(device.Get());

// Release を明示的に呼ぶ必要はない（スコープ終了時に自動解放）

    実際の開発では ComPtr の利用が推奨される。

    ただし、本資料のサンプルコードでは「各オブジェクトが生ポインタである」ことを明示し、
    生ポインタと Release() の関係を理解しやすくするため、あえて生ポインタで統一している。
    実装の際には ComPtr への置き換えを検討してほしい。

    おまけ：デバッグレイヤーの有効化

    開発段階では、DirectX12 のデバッグレイヤーを有効にしておいた方が良い。
    これは「開発者向けの親切な先生」のような存在で、コードの問題点やパフォーマンスの改善点を詳細に教えてくれる。

    void EnableDebugLayer()
{
#if defined(_DEBUG)
    // デバッグインターフェースを取得
    ID3D12Debug* debugController;
    if (SUCCEEDED(D3D12GetDebugInterface(IID_PPV_ARGS(&debugController))))
    {
        // デバッグレイヤーを有効化
        debugController->EnableDebugLayer();

        // より詳細な検証を有効化（任意）
        ID3D12Debug1* debugController1;
        if (SUCCEEDED(debugController->QueryInterface(IID_PPV_ARGS(&debugController1))))
        {
            debugController1->SetEnableGPUBasedValidation(TRUE);
        }
    }
#endif
}

    デバッグレイヤーの効果：

    • メモリリークの検出
    • 不正なAPI使用の警告
    • パフォーマンス問題の指摘
    • リソース状態の不整合検出

    デバッグレイヤーを有効にすると、Visual Studioの出力ウィンドウに詳細な情報が表示される。
    初心者はこの情報を頼りにコードを修正していく。

    この検証により、次回講義のレンダリングパイプライン構築に向けた基盤が整ったことが確認できる。
    各オブジェクトが正常に作成されていれば、GPU上での描画処理が実行可能な状態である。
