    レンダリングパイプライン

    ゲーム開発において、グラフィックスの表現は作品の質に大きく関係してくる。
    3Dオブジェクトがどのようにして画面に描画されるのか、その仕組みを理解する事は、
    描画を担当しない場合でも、ゲームエンジニアとして活躍する為にある程度必要になる。

    また就職を目指す意味でも、Unity や UE ではなく、
    自身で描画処理をゼロから作る経験はしておいてほしい。

    レンダリングパイプラインとは

    基本概念

    レンダリングパイプラインとは、3Dデータを2D画面に描画するための一連の処理工程のこと。
    まるで工場の製造ラインのように、段階的に処理が進んでいくことからパイプラインと呼ばれている。

    3D空間に配置されたモデルデータが、
    最終的にモニター上のピクセルとして表示されるまでの変換プロセス全体を指す**

    CPU と GPU の役割分担

    これまで作成してきた Unity プログラムは全て CPU で動作している。
    逆に言えば、これまでは CPU しか使わないプログラムしかやってきていない。
    ところが描画処理を扱う場合は、CPU だけではなく GPU も扱う必要が出てくる。

    まず、これらプロセッサの働きをイメージ出来る事が大事。

    CPU（Central Processing Unit）の役割

    CPUは「コンピュータの頭脳」で、少数のコア（8〜16コアあたりが一般的）で、
    複雑な処理を順次実行するのが得意。
    CPU の具体的な役割は以下の通り。

    ゲームロジックとシミュレーション

    所謂「ゲームの挙動」全般を担当している。

    • キャラクターの移動、回転、アニメーション制御
    • 物理演算（重力、衝突判定、剛体シミュレーション）
    • AI の思考ルーチン、パスファインディング
    • ゲームルールの管理、スコア計算

    描画データの準備と管理

    描画する為に必要な情報の準備を行う。

    • 3Dモデルの頂点データ、インデックスデータの管理
    • テクスチャファイルの読み込みと前処理
    • マテリアル情報の整理
    • 変換行列（ワールド行列、ビュー行列、プロジェクション行列）の計算

    描画設定の管理

    GPU に対して「何を」「どのように」描画するかを指示する。

    • 描画すべきオブジェクトの選別と優先順位付け
    • 各オブジェクトに適用するシェーダプログラムの選択
    • テクスチャやマテリアル設定の組み合わせ決定

    GPU への描画指示

    GPU に実行させる命令の一覧を作り、実際に実行させる。
    CPU は GPU に実行してもらいたい処理を「コマンドバッファ」という形で纏めている。

    描画設定の纏め方

    1. レンダーターゲットの設定: どの画面（バックバッファ）に描画するか
    2. ビューポートの設定: 画面のどの領域に描画するか（全画面、ウィンドウの一部など）
    3. シェーダプログラムの指定: どのVertex Shader、Pixel Shader を使用するか
    4. 入力レイアウトの設定: 頂点データの構造（位置、色、法線などの配置）を GPU に伝達

    リソースの結び付け

    • テクスチャスロットへのテクスチャ割り当て（「スロット0番には草のテクスチャ、スロット1番には法線マップ」など）
    • 定数バッファの結び付け（カメラ行列、ライト情報などをシェーダで使えるように設定）
    • サンプラーステートの設定（テクスチャをどのようにサンプリングするかの指定）

    実際の描画命令

    // 擬似コード例（実際のDirectXコード風）
1. SetRenderTarget(backBuffer)           // 描画先を指定
2. SetViewport(0, 0, 1920, 1080)        // 描画領域を指定  
3. SetVertexShader(characterVS)          // 頂点シェーダを設定
4. SetPixelShader(characterPS)           // ピクセルシェーダを設定
5. SetTexture(0, characterTexture)       // テクスチャを結び付け
6. SetVertexBuffer(characterVertices)    // 頂点データを設定
7. SetIndexBuffer(characterIndices)      // インデックスデータを設定
8. DrawIndexed(triangleCount)            // ここまで設定した内容を元に、実際に描画実行命令

    GPU（Graphics Processing Unit）の役割

    GPUは数百から数千の小さなコアを持ち、
    同じ処理を大量のデータに対して並列実行する事に特化している。
    この特性がグラフィック処理に非常に適している。

    大量並列処理の仕組み

    例えば、1920×1080の画面には約200万個のピクセルが存在する。
    GPU は各ピクセルの色計算を同時並行で実行できるため、リアルタイムでの描画が可能になる。

    頂点処理（Vertex Processing）

    • 3D座標から2D座標への変換計算
    • 頂点ごとの法線ベクトル計算
    • テクスチャ座標の変換
    • 頂点カラーの計算

    ピクセル処理（Pixel Processing）

    • 各ピクセルの最終色の決定
    • テクスチャサンプリング（テクスチャから色を取得）
    • ライティング計算（光の当たり方による明暗）
    • 特殊効果の適用（反射、屈折、パーティクルなど）

    CPUからGPUへのデータ配置の詳細な流れ

    PC には「CPU用のメモリ（RAM）」と「GPU用のメモリ（VRAM）」という、
    それぞれ別々のデータ置き場（メモリ領域）が存在する。

    例えばゲームのキャラクターや背景のデータは、まず CPU が RAM に読み込み、
    その後 GPU が「高速に」アクセス出来るように VRAM へコピーする。

    GPU が利用するデータは、必ずしも VRAM に配置しなければならないわけではない。
    「高速に処理する為に使い分ける」という点で VRAM は重要になる。

    最近はストレージ（SSD/HDD）から直接 VRAM にデータを送る技術（DirectStorage）も登場している。

    メモリ種類

    メモリ種類 アクセス可能な処理装置 容量 特徴
    システムメモリ（RAM） CPU (GPU) 8GB〜64GB程度 ゲームデータ、テクスチャファイル、モデルデータなどを保存
    グラフィックスメモリ（VRAM） GPU 4GB〜24GB程度 桁違いな高速アクセスが可能、頂点バッファ、テクスチャ、シェーダプログラムなどを保存

    システムメモリでのデータ準備段階

    ファイルからシステムメモリへの読み込み

    CPU は最初にストレージ（SSD/HDD）からゲームデータをシステムメモリに読み込む。

    • 3Dモデルファイル：.obj、.fbx、などの形式で保存された頂点データ
    • テクスチャファイル：.png、.jpg、.dds などの形式で保存された画像データ
    • シェーダファイル：.hlsl で記述されたシェーダソースコード

    データの前処理と最適化

    システムメモリ上のデータを GPU で処理しやすい形に変換する。

    • 頂点データの並び替え：GPU が効率的に処理できる順序に頂点を配置
    • シェーダのコンパイル：HLSL ソースコードを GPU が実行できるバイナリ形式に変換

    各データを GPU で利用可能にする

    CPU で各データの準備を終えても、システムメモリ上にあるだけでは描画に利用できない。
    それらデータを GPU で利用できるように手続きが必要になる。

    DirectX では GPU が使うメモリをヒープと呼ぶ。
    代表的には 3 種類あり、それぞれ特性が違う。
    その特性に合わせて、使い分ける必要がある。

    メモリヒープの種類と配置場所

    DirectXでは、用途に応じて異なるメモリヒープタイプを使用する。

    ヒープタイプ 物理的な配置場所 GPUのアクセス方法 主な用途
    UPLOAD システムメモリ PCIeバス経由で直接読み込み CPU→GPU 頻繁に更新されるデータ
    DEFAULT VRAM VRAM上に配置、高速アクセス GPU専用の静的データ
    READBACK システムメモリ GPUからCPUへの書き戻し 計算結果の読み戻し

    データ配置の実行方法

    CPU は DirectX API を使用して、準備したデータを GPU で利用可能な状態にする。

    Map/Unmap による方法

    • CPU がアクセス可能なメモリ領域（UPLOADヒープ）へのポインタを取得（Map）
    • その領域に直接データを書き込み
    • このメモリはシステムメモリ上に存在し、GPU が直接読み込む
    • 必ずしも VRAM にコピーされるわけではなく GPU が読める場所にデータを置く
    • 頻繁に更新される定数バッファ、動的頂点データなどに使用

    UpdateSubresource/CopyResource による方法

    • システムメモリから VRAM 上の DEFAULT ヒープへデータコピーを実行
    • データは VRAM 上に配置され、GPU から超高速にアクセス可能
    • DirectX ドライバが内部で最適な転送方法を選択
    • 静的なモデルデータ、テクスチャ、一度作成したリソースの更新などに使用

    具体例：キャラクターモデルの場合

    Step 1：システムメモリでの準備

    // システムメモリ上のデータ構造例
struct Vertex {
    float position[3];    // 頂点座標（x,y,z）
    float normal[3];      // 法線ベクトル
    float texCoord[2];    // テクスチャ座標（u,v）
};

Vertex* vertices = new Vertex[vertexCount];
uint32_t* indices = new uint32_t[indexCount];
// ファイルから読み込んだデータで配列を埋める

    Step 2：GPUバッファの作成とデータ配置

    // DirectX を通じてGPUが利用可能なバッファを作成
ID3D12Resource* vertexBuffer;
ID3D12Resource* indexBuffer;

// 静的データの場合：作成と同時にVRAMに配置
CreateBufferAndInitialize(&vertexBuffer, DEFAULT_HEAP, vertices, vertexDataSize);
CreateBufferAndInitialize(&indexBuffer, DEFAULT_HEAP, indices, indexDataSize);

// 動的データの場合：UPLOADヒープを使用
CreateBuffer(&dynamicBuffer, UPLOAD_HEAP, dynamicDataSize);
// Map/Unmapで毎フレーム更新

    Step 3：描画時のバッファ結び付け

    // 描画時にGPUにバッファの場所を指定
SetVertexBuffers(0, 1, &vertexBuffer);
SetIndexBuffer(indexBuffer);
SetConstantBuffer(0, &dynamicBuffer);  // 動的データ

    この手続きにより、CPU で準備されたデータが GPU で利用可能となり、
    実際の描画処理が実行できるようになる。
    静的なデータ（背景や動かないモデルなど）は一度転送すれば、
    以降は GPU が独立して高速な描画処理を継続できる。

    一方、キャラクターの動きやアニメーションなど動的なデータは、
    必要に応じて毎フレーム転送や更新が必要になってくる。

    レンダリングパイプラインの歴史

    固定機能パイプライン時代（1990年代後半〜2000年代前半）

    初期のグラフィックスハードウェアでは、レンダリングパイプラインの各段階が
    固定的に決められていた。
    開発者はパラメータを調整することしかできず、表現の自由度は限られていた。

    プログラマブルシェーダの登場（2000年代）

    DirectX 8.0（2000年）でVertex Shaderが、DirectX 9.0（2002年）でPixel Shaderが導入された。
    これにより、開発者が独自のシェーディング処理を記述できるようになり、
    表現の幅が大きく広がった。

    統合シェーダアーキテクチャ（2006年〜）

    DirectX 10以降では、より様々な機能が追加されたシェーダモデルが採用され、
    現在に至るまでの基礎となっている。

    DirectXにおけるレンダリングパイプラインの流れ

    DirectXのレンダリングパイプラインは、以下の段階で構成されている。

    Input Assembler（入力組立）段階

    CPU から送られてきた頂点データ（位置、法線、テクスチャ座標など）を読み込み、
    プリミティブ（三角形など）を構成する。

    Vertex Shader（頂点シェーダ）段階

    各頂点に対して実行される処理。主に以下の変換を行う。

    • ローカル座標からワールド座標への変換
    • ワールド座標からビュー座標への変換
    • ビュー座標からクリップ座標への変換

    Hull Shader / Domain Shader（テセレーション）段階

    より滑らかな曲面を生成するために、既存のプリミティブを細分化する。
    この段階は必須ではない。

    Geometry Shader（ジオメトリシェーダ）段階

    プリミティブ単位での処理を行う。
    頂点の追加や削除、プリミティブの種類変更などが可能である。

    Rasterizer（ラスタライザ）段階

    3Dプリミティブを2Dピクセルに変換する。
    また、画面外のプリミティブを除去するクリッピング処理も行う。

    Pixel Shader（ピクセルシェーダ）段階

    各ピクセルの最終的な色を計算する。
    テクスチャサンプリング、ライティング計算、各種エフェクトの適用などを行う。

    Output Merger（出力マージャー）段階

    最終的な色をフレームバッファに書き込む。
    深度テスト、アルファブレンディングなどの処理も含まれる。

    シェーダとは

    シェーダの基本概念

    シェーダとは、GPU 上で実行される小さなプログラム。
    「シェーディング（陰影付け）」という名前の通り、元々は物体の明暗や色を計算するためのものであったが、
    現在では様々な視覚効果を実現するための汎用的なプログラムとして使われている。

    シェーダの特徴

    • GPU の並列処理能力を活用
    • 大量のデータを同時に処理
    • リアルタイムでの高速な計算が可能

    DirectXにおけるシェーダ言語

    DirectXでは、HLSL（High Level Shading Language）という専用言語でシェーダを記述する。
    C言語に似た文法で、比較的習得しやすい。

    主なシェーダの種類

    Vertex Shader（頂点シェーダ）

    3D空間の頂点位置を2D画面座標に変換する。
    モデルの変形、カメラ視点の反映、頂点ごとのライティング計算などを行う。

    Pixel Shader / Fragment Shader（ピクセルシェーダ）

    各ピクセルの最終的な色を決定する。
    テクスチャの適用、詳細なライティング、特殊効果などを実装する。
    現代のゲームグラフィックスのリッチ表現は、主にこのシェーダの技術によるもの。

    Geometry Shader（ジオメトリシェーダ）

    プリミティブレベルでの操作を行う。
    パーティクル生成、影の描画、ワイヤーフレーム表示などに活用される。

    Compute Shader（コンピュートシェーダ）

    レンダリングパイプラインとは独立して動作し、汎用的な並列計算を行う。
    物理シミュレーション、画像処理、AI計算などに使用される。

    現代のレンダリングパイプラインの課題と展望

    パフォーマンスの最適化

    モバイルデバイスでも高品質なグラフィックスが求められる現在、
    効率的なレンダリングパイプラインの設計がより重要になっている。

    リアルタイムレイトレーシング

    従来のラスタライゼーションベースのパイプラインに加え、
    よりリアルな光の表現を可能にするレイトレーシング技術が実用化されつつある。

    機械学習の活用

    AI技術を活用したアップスケーリングや画質向上技術（DLSS、FSRなど）も、
    現代のレンダリングパイプラインに組み込まれるようになっている。

    新しいレンダリングパイプライン

    コンピュートシェーダの併用が一般化してきている事もあり、
    より高速にハイモデル描画を実現する MeshShader も一般化しつつある。
    また、更に高速な描画処理を目指し、 Work Graph といった試験的な機能の開発も進んでいる。

    まとめ

    レンダリングパイプラインは、3Dゲーム開発における
    基礎的かつ重要な概念。
    CPU と GPU の動作と連携、各段階での処理内容、シェーダの役割を理解することで、
    より高品質なゲーム開発が可能になる。

    技術は日々進歩しているが、基本的な概念や処理の流れは変わらない。
    まずはこの基礎をしっかりと理解し、実際のDirectX APIを使った開発経験を積むことで、
    ゲーム開発者として必要なグラフィックスプログラミングスキルを身につけていくことができる。
